# 术语热词库接入会议翻译链路设计

> 版本：v0.1\
> 日期：2026-06-23\
> 目标：在不破坏当前实时会议翻译链路速度与稳定性的前提下，把“固定译法”型术语热词库真实接入会中 segment 翻译流程。

## 一、设计结论

本轮第六步的目标，不是重写现有翻译模块，而是在当前链路上增加一个**极轻量的术语命中分支**：

- segment 原文落成后，先按当前 `sourceLanguage -> targetLanguage` 去术语库检索
- 未命中：继续走当前翻译链路，prompt 不变
- 命中：进入“术语增强翻译路由”
- 术语增强翻译路由仍然调用同一个翻译模型，不新增模型能力
- 术语增强路由只比原链路多两件事：
  - `system prompt` 增加极少量术语约束说明
  - `user prompt` 增加 `glossary_terms` 字段

本轮只把**固定译法**接入真实翻译链路：

- `固定译法`：真实命中、真实传入 prompt、真实影响译文
- `识别热词`：本轮不接 ASR，不进入本次翻译 prompt 主流程

## 二、当前代码基线

### 1. 现有 segment 翻译链路

当前会中翻译链路已经存在，关键路径如下：

1. `meeting-page-app.js` 收到 ASR 返回的原始 segment
2. `createSegmentRecord()` 中调用 `detectLanguage()` 判定 `sourceLanguage`
3. 调用 `createTranslationPlan()` 决定 `desiredTargetLanguage`
4. 如果需要翻译，进入 `requestTranslation(record)`
5. `requestTranslation()` 调用 `translateSegment()`
6. `meeting-translation-executor.js` 构造 prompt 并请求 `senseaudio-s2-lite`
7. 译文回填页面、PiP 和 transcript snapshot

### 2. 现有翻译执行器特征

当前执行器已经具备：

- 常规翻译路由
- 超短句专用路由 `short_utterance`
- `contextTail` 上下文尾部传入
- `temperature: 0`
- 输出仅包含译文

这意味着：当前翻译器已经是“**system prompt + user prompt 字符串拼接**”的结构，非常适合在此基础上追加术语块。

### 3. 现有术语存储侧能力

当前术语 store 已经具备：

- `findMatchingTermRules(payload)`
- `buildGlossaryPromptTerms(matches)`
- `recordMatchedTermHits(matches)`

并且已实现以下运行时规则：

- 中文 / 日文按 `contains`
- 英文按 `whole_word`
- 企业共享优先于个人术语
- 同术语同语言对只允许一条规则
- 多条命中时会做冲突裁剪

因此第六步无需重做术语检索引擎，只需要把 store 的命中结果接到会中翻译入口。

## 三、本轮接入边界

### 1. 本轮接什么

本轮接入范围：

- `固定译法`
- `segment` 级命中
- prompt 增强
- 命中计数

### 2. 本轮不接什么

本轮不接入：

- ASR 热词参数注入
- `识别热词` 的真实识别增强
- 大规模 prompt 重写
- 跨段长上下文术语推理
- 用户可配置的高级匹配模式

## 四、目标链路

推荐在 `meeting-page-app.js` 中新增一个极小的术语接入层，逻辑如下：

```text
ASR segment
  -> detectLanguage()
  -> createTranslationPlan()
  -> if shouldTranslate === false: 结束
  -> matchTermsForSegment()
  -> if glossaryTerms.length === 0:
       translateSegment(base route)
     else:
       translateSegment(glossary route)
  -> 回填译文
  -> 命中计数
```

## 五、模块拆分建议

### 1. 新增模块

建议新增：

`src/meeting/meeting-terms-matcher.js`

职责：

- 接收 segment 文本与语言对
- 调用 `findMatchingTermRules()`
- 将命中结果转换为可直接喂给翻译执行器的 `glossaryTerms`
- 返回命中摘要，供日志与命中计数使用

建议暴露接口：

```js
export function matchGlossaryForSegment({
  actor,
  text,
  sourceLanguage,
  targetLanguage,
}) {
  // return {
  //   matches,
  //   glossaryTerms,
  //   hasGlossary,
  // }
}
```

### 2. 保持执行器单一入口

不建议再新开一个完全独立的 `translateSegmentWithGlossary()`。

更稳的方式是：

- 继续保留 `translateSegment()` 作为唯一入口
- 给它增加可选参数：

```js
translateSegment({
  apiKey,
  model,
  sourceLanguage,
  targetLanguage,
  text,
  contextTail,
  glossaryTerms = [],
})
```

这样：

- 未命中术语时，`glossaryTerms` 为空数组
- 命中术语时，执行器自动切到 glossary route

## 六、接口设计

### 1. `meeting-page-app.js` -> `meeting-terms-matcher.js`

输入：

```js

  actor,
  text,
  sourceLanguage,
  targetLanguage,
}
```

输出：

```js
{
  matches: [
    {
      entryId,

  glossaryTerms: [

  hasGlossary: true
}
```

说明：

- `matches` 用于命中计数
- `glossaryTerms` 用于拼 prompt
- `hasGlossary` 用于路由判断

### 2. `meeting-page-app.js` -> `meeting-translation-executor.js`

新增入参：

```js
{
  glossaryTerms: [],
}
```

### 3. `meeting-translation-executor.js` 内部输出

建议保留现有返回结构，只新增少量调试字段：

```js
{
  translatedText,
  model,
  latencyMs,
  status,
  glossaryApplied: true,
  glossaryTermCount: 3,
  raw,
}
```

这些字段主要用于调试和日志，不要求页面必须展示。

## 七、命中逻辑

### 1. 只按当前语言对检索固定译法

运行时检索条件：

- `termType = fixed_translation`
- `sourceLanguage = 当前 segment 识别语种`
- `targetLanguage = 当前 translationPlan.desiredTargetLanguage`

例如：

- 当前 segment 被判定为 `zh`
- 当前目标语言为 `ja`
- 则只查 `zh -> ja`

### 2. 命中方式

命中方式沿用当前 store 的默认策略：

- 中文 / 日文：`contains`
- 英文：`whole_word`

即：

- 中文 / 日文句子里只要出现该术语片段，就算命中
- 英文只有在句子中作为完整单词出现，才算命中

### 3. 命中结果数量控制

本轮按业务优先处理：

- 命中多少条，就传多少条
- 不在 matcher 层做人为截断
- 不因为“超过 5 条”而忽略客户明确配置的术语

说明：

- 本轮接受翻译耗时轻微变慢
- 如果后续真实线上数据证明 prompt 过长明显影响体验，再基于监控结果做优化

### 4. 企业优先

如果同一个 segment 同时命中企业和个人术语：

- 以企业共享为优先
- 个人术语只作为补充

这部分已由 store 层排序与冲突裁剪承担，本轮不在会中侧重复实现。

## 八、路由设计

### 1. 基础路由

当且仅当 `glossaryTerms.length === 0` 时：

- 继续走当前翻译路由
- `system prompt` 不变
- `user prompt` 不变

### 2. 术语增强路由

当 `glossaryTerms.length > 0` 时：

- 走 glossary route
- 仍然调用同一个 `translateSegment()`
- 只在 prompt 层增加术语字段

### 3. 与短句路由的关系

本轮建议：

- 短句仍然保留 `short_utterance` 的基础 prompt 风格
- 但只要命中术语，短句也要带上 `glossary_terms`

原因：

- 客户明确配置的术语，不应因为短句路由而被跳过
- 短句场景仍然可以不带 `context_tail`，从而保持整体 prompt 足够短

因此更准确的实现方式是：

```text
base_route / short_utterance_route
  + glossary_terms(仅在命中时追加)
```

## 九、Prompt 设计

### 1. 原则

你提出的要求是对的：

- 原始 prompt 尽量不动
- 只在命中术语时追加极少量约束
- 要告诉模型“参考术语翻译”
- 同时提醒它保持整体通顺，不要机械硬替换

本轮的 prompt 目标不是“强制硬替换”，而是：

- **优先参考术语**
- **但仍保持整句自然、符合上下文**

### 2. glossary route 的 system prompt 建议

建议在现有 `SYSTEM_PROMPT` 基础上，拼接一段极短的术语说明：

```text
以下 glossary_terms 是本场会议的术语热词。
翻译时应优先参考这些术语的对应译法或保留原文要求。
如果某个术语命中与当前语义明显不符，不要生硬套用；请在保持原意与整句通顺的前提下自然翻译。
输出仍然只包含译文，不要附加说明。
```

说明：

- 第一条：明确 glossary_terms 的作用
- 第二条：明确优先参考
- 第三条：给模型一个“避免误命中硬套”的缓冲
- 第四条：维持当前输出格式

这段已经足够，不建议再加更多规则。

### 3. glossary route 的 user prompt 建议

原来的 user prompt 保留，只增加一个字段块：

```text
source_language: zh
target_language: ja
context_tail:
- ...
- ...
glossary_terms:
- 源词: 神泉科技 | 处理方式: 固定翻译 | 译法: シェンチュアン
- 源词: APPI | 处理方式: 固定翻译 | 译法: 個人情報保護法
- 源词: 田中健太 | 处理方式: 保留原文 | 译法:
text: 今天我们和神泉科技讨论 APPI 合规，也提到了田中健太负责的项目。
```

本轮采用更短的第一种，不使用更长的半结构化格式。

### 4. `buildUserPrompt()` 的建议改造

建议新增入参：

```js
buildUserPrompt({
  sourceLanguage,
  targetLanguage,
  contextTail,
  text,
  useShortUtteranceRoute,
  glossaryTerms = [],
})
```

行为：

- `short_utterance`：
  - 不带 `context_tail`
  - 命中术语时仍拼接 `glossary_terms`
- 普通路由：
  - 无 glossary：保持原样
  - 有 glossary：在 `context_tail` 后、`text` 前增加 `glossary_terms`

## 十、误命中防护

你提到的“黑夜总会过去 -> 夜总会”这类风险是成立的。

本轮建议采用“双层防护”：

### 1. 第一层：匹配层防护

- 中文 / 日文：仍按 contains
- 英文：按 whole_word
- 由 store 侧提前做冲突裁剪
- 不做“只保留前 5 条”的人为截断

### 2. 第二层：大模型自然校正

在 system prompt 明确告诉模型：

- 术语应优先参考
- 但如果与上下文语义明显不符，不要生硬套用
- 保持整体自然通顺

这就是你想要的“参考、意译、保持通顺”的能力来源。

注意：

- 这不是把判断完全甩给模型
- 而是在“轻量命中 + 术语提示”之后，给模型保留少量语义纠偏空间

对于 MVP，这样是合适的。

## 十一、命中计数

建议在 glossary route 成功返回后，再记录命中计数：

- 只有成功发起术语增强翻译并成功返回译文，才记一次命中
- 如果翻译请求失败，不记命中

调用方式：

```js
recordMatchedTermHits(matches)
```

放置位置：

- `requestTranslation()` 成功分支
- 且仅在 `matches.length > 0` 时调用

## 十二、记录结构建议

建议给 segment record 增加以下字段：

```js
{
  glossaryMatches: [],
  glossaryTerms: [],
  glossaryApplied: false,
}
```

用途：

- 页面调试
- 后续命中统计
- 未来如果要在 transcript 中显示“术语命中”标识，也有数据基础

## 十三、开发步骤建议

### 1. 新增 matcher

- 新建 `src/meeting/meeting-terms-matcher.js`
- 接好 `findMatchingTermRules()` 与 `buildGlossaryPromptTerms()`

### 2. 修改 record 结构

- 在 `createSegmentRecord()` 里初始化 glossary 相关字段

### 3. 修改请求入口

- 在 `requestTranslation(record)` 开始前做术语匹配
- 命中则把 `glossaryTerms` 带入执行器

### 4. 修改执行器

- `translateSegment()` 增加 `glossaryTerms`
- `buildUserPrompt()` 支持 `glossary_terms`
- `system prompt` 在 glossary route 下追加术语说明

### 5. 修改成功回调

- glossary 命中成功后调用 `recordMatchedTermHits()`

## 十四、验收标准

### 1. 无命中场景

- 翻译结果与当前链路保持一致
- 不新增明显延迟

### 2. 固定译法命中场景

- 命中的术语会被带入 glossary route
- 译文优先采用术语库中的指定写法

### 3. 保留原文场景

- 命中的术语在译文中保持原文，不被错误翻译

### 4. 误命中缓冲场景

- 即便存在低风险误命中，整体句子仍应保持自然
- 不应出现明显的机械替换和强行生搬硬套

### 5. 联动结果

- transcript 译文更新正确
- PiP 悬浮字幕同步正确
- 后续 AI 助手和总结读取到的是同一份统一术语后的译文

## 十五、最终建议

本轮第六步最稳的做法是：

- 不改动现有翻译主链路结构
- 不增加新的模型
- 不大幅扩展 prompt
- 只在“命中固定译法术语”时，追加一个极小的 glossary route

这能同时满足：

- 真实功能接入
- 速度尽量不受影响
- 术语翻译更稳定
- 对误命中保留一定模型纠偏空间

这也是最符合当前 MVP 目标的实现方式。
