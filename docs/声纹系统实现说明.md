# 声纹识别系统实现说明

本文介绍本项目「会议中自动识别说话人」功能的完整实现：从声音到身份，端到端在浏览器里完成，声纹数据不出设备。

> 适用范围：本文描述当前已落地的代码（`src/integrations/voiceprint-engine.js`、`src/storage/voiceprint-store.js`、`app/voiceprint.html` 注册流程、`src/meeting/meeting-page-app.js` 会议接入）。算法参数若有调整，以代码为准。

---

## 1. 一句话概括

用一个开源的中文声纹模型（CAM++），在浏览器里把每个人的声音压缩成一串 192 个数字（声纹向量）。注册时把你的声音存成这串数字；开会时把每句话的声音也转成这串数字，和库里的逐一比对，最像谁就标谁的名字。

整条链路——录音、特征提取、模型推理、比对——全部在浏览器本地运行，声纹向量只存在你浏览器的 IndexedDB 里，不上传服务器。

---

## 2. 核心原理：声纹 = 一段声音的“指纹向量”

声纹识别的本质是把任意长度的一段语音，映射成一个固定长度（这里是 192 维）的向量。同一个人的不同句子，得到的向量彼此很接近；不同人的向量则相距较远。“接近”用**余弦相似度**衡量（两个向量夹角的余弦，范围 -1~1，越接近 1 越像）。

所以识别就两步：
1. **注册**：录你几秒声音 → 算出你的声纹向量 → 存库。
2. **识别**：会议中每句话 → 算出该句的声纹向量 → 和库里每个人的向量算余弦 → 超过阈值就判定为那个人，否则“未识别”。

这套方案不依赖说话内容（说什么都行），只看声音特质，所以中英文混说也不影响。

---

## 3. 模型：CAM++

- 模型文件：`app/models/campplus_cn_common.onnx`（约 29.7MB）
- 来源：达摩院开源的 CAM++ 中文通用声纹模型，原始为 PyTorch 权重，已导出为 ONNX 格式供浏览器使用。
- 输入：FBank 声学特征，形状 `[1, 帧数, 80]`
- 输出：192 维声纹向量 `[1, 192]`

### 一个踩过的坑：外置权重

模型最初导出时，ONNX 把 27MB 的权重单独存成了 `campplus_cn_common.onnx.data`（外置数据文件），主文件只剩 2.1MB 的计算图。**浏览器端的 onnxruntime-web 不支持加载这种外置权重**，会报 `Failed to load external data file`。

解决办法是重新导出成**单文件内嵌**版本（用 `onnx.load` 把权重读进来后，`onnx.save` 不带外置选项另存），得到 29.7MB 的单文件，并验证内嵌前后模型输出余弦相似度为 0.99999994（完全一致）。现在 `app/models/` 下就是这个内嵌单文件。

### 推理引擎：onnxruntime-web

通过 CDN 动态加载 onnxruntime-web 1.20.1（纯 ESM，无需打包工具），执行后端用 wasm。模型在浏览器里跑推理，首次加载需从 CDN 下 wasm 运行时 + 本地下 30MB 模型，之后缓存。

## 4. 特征提取：FBank（最硬的一关）

模型不直接吃原始音频波形，而是吃 **FBank 特征**（80 维 Mel 滤波器组能量）。这一步必须和模型训练时用的特征算法**完全一致**，否则模型“听不懂”，识别全错。

CAM++ 训练时用的是 **Kaldi 风格的 FBank**。问题是 Kaldi 是 C++ 工具，浏览器里没有，所以我们用纯 JavaScript **从零复刻**了一套 Kaldi 兼容的 FBank（见 `voiceprint-engine.js` 的 `computeFbank`）。复刻的每个环节都要对齐 Kaldi 的默认行为：

| 环节 | 参数/做法 | 说明 |
|------|-----------|------|
| 采样率 | 16000 Hz | 模型要求 |
| 帧长 / 帧移 | 25ms / 10ms（400 / 160 样本） | Kaldi 默认 |
| 输入量级 | PCM 乘 32768 | Kaldi 按 int16 量级处理 |
| 去直流 | 每帧减均值 | |
| 预加重 | 系数 0.97 | |
| 加窗 | povey 窗（`(0.5-0.5cos)^0.85`） | Kaldi 默认窗，不是常见的 Hamming |
| FFT | 512 点（≥400 的最近 2 的幂），自实现 Cooley-Tukey | |
| Mel 滤波 | 80 个三角滤波器，频率范围 20Hz~8000Hz，hz↔mel 用 1127·ln 公式 | |
| 取对数 | `log(max(能量, EPSILON))` | EPSILON 防 log(0) |
| CMN | 实例级倒谱均值归一化（沿时间轴每维减均值） | |

### 如何确认复刻是对的：对拍验证

这是整个项目风险最高的一关。验证方法（`models/_export/verify_fbank.py`）：用同一段音频，分别喂给

- 我们的 JS 版 FBank
- 官方 torchaudio 的 Kaldi FBank

再把两套特征分别送进同一个声纹模型，比较输出的声纹向量。结果：

- 特征级最大差异 0.0115，平均差异 0.0015（来自 FFT 浮点实现细节，可忽略）
- **最终声纹向量余弦相似度 0.99999**

这个结果证明 JS 复刻的 FBank 和官方实现等价，模型能正确“听懂”浏览器算出来的特征。这一关过了，方案才成立。

## 5. 从音频到声纹向量的完整流程

`extractEmbedding(pcm)` 做的事（输入是 Float32、[-1,1]、16kHz 的 PCM）：

```
PCM 波形
  → computeFbank()         算出 [帧数, 80] 的 FBank 特征
  → 帧数 < 10 则放弃        太短的音频不可靠
  → ONNX 推理 session.run  得到 192 维原始向量
  → l2normalize()          L2 归一化（向量除以自身模长）
  → 192 维单位向量
```

L2 归一化很关键：归一化后，两个向量的余弦相似度等于它们的点积，比对时直接点积即可，又快又准。

## 6. 声纹库存储（`voiceprint-store.js`）

声纹存在浏览器的 **IndexedDB**（数据库名 `ac_voiceprints`），不是 localStorage——因为每条声纹是 192 个浮点数，IndexedDB 更适合存这种结构化二进制数据。

每条声纹记录的结构：

```js
{
  id,            // vp_<时间戳>_<随机>
  ownerUserId,   // 归属用户 id —— 声纹按账号隔离
  name,          // 姓名
  role,          // 角色（可选）
  color,         // 气泡配色（从调色板按序分配）
  avatar,        // 头像首字
  embedding,     // 192 维声纹向量（已归一化）
  sampleCount,   // 注册用了几段音频
  createdAt, updatedAt
}
```

**按账号隔离**：每条记录带 `ownerUserId`，并建了索引。`getVoiceprintsForCurrentUser()` 只返回当前登录账号的声纹。这意味着如果注册声纹和开会时登录的不是同一个账号，会读不到声纹——这是之前调试时遇到过的一个坑。

主要 API：`getVoiceprintsForCurrentUser`、`addVoiceprintForCurrentUser`、`deleteVoiceprint`、`renameVoiceprint`。

## 7. 注册流程（`app/voiceprint.html`）

用户在声纹管理页点「添加声纹」后：

1. 填姓名、角色。
2. 点录音区，用麦克风录约 10 秒（`AudioPcmCapture` 采集，自动降采样到 16kHz、转成 Int16 PCM）。
3. 把录到的音频提取声纹向量。为提高稳定性，注册时会把录音切成多段、每段各算一个向量，再用 `averageEmbeddings` **取平均后归一化**——多段平均能抵消单段里的偶然噪声，得到更稳的声纹。
4. 勾选两个同意项（隐私确认）后，存入 IndexedDB。
5. 列表实时刷新，显示新成员。

页面列表完全由 IndexedDB 真实数据渲染（库空时显示空状态，不再是写死的演示人员）。

## 8. 会议中实时识别（`meeting-page-app.js`）

这是最复杂的一环，难点在于：ASR（语音转写）和声纹识别要并行，但识别不能拖慢转写显示。设计如下：

### 8.1 会议开始时

异步预热声纹引擎 + 从 IndexedDB 加载当前账号的声纹库（`prepareVoiceprint`）。这一步不阻塞会议启动。加载完成后状态栏提示「已加载 N 个声纹」。**若声纹库为空，自动回退到原来的演示发言人逻辑**，不破坏未注册时的体验。

### 8.2 音频环形缓冲

麦克风采集的每块 PCM，一边发给 ASR 转写，一边写入一个**30 秒环形缓冲**（`pushVoiceBuffer`）。环形的意思是缓冲满了就回绕覆盖最旧的，始终保留最近 30 秒。用一个单调递增的总样本计数 `vpRingFilled` 配合取模，既能定位又能判断某段音频是否已被覆盖。

### 8.3 每句话的识别（先显示，后回填）

ASR 每吐出一句最终结果时：

1. **立即**用占位发言人建气泡，把转写文字显示出来（不等声纹，保证转写不卡）。
2. **异步**触发 `identifySpeakerForRecord`：从环形缓冲取出这句话的音频 → 提声纹 → 和声纹库比对 → 拿到身份后**回填**气泡的姓名/头像/配色。

这种“先占位、后回填”的设计让文字秒显，身份一两秒后补上，体验流畅。

### 8.4 取哪一段音频（关键设计）

最初的实现按 ASR 给的绝对时间戳从缓冲里切音频，但有个隐患：**ASR 的时间戳从转写任务启动计时，而音频缓冲从麦克风启动计时，两个时钟起点不同**，导致切出的音频系统性偏移，可能切到上一句或静音——表现为识别“时对时错、认成别人、未识别”三种问题混杂。

当前实现改为**锚定缓冲末尾**：一句话刚说完时，缓冲尾部就是这句的音频，于是直接取“最近的一段”（`sliceRecentVoice`），窗口长度用 ASR 的句长（两个时间戳**相减**得到，同源相减能抵消起点偏移，所以可靠）自适应，夹在 1~8 秒之间。这样绕开了时钟对齐问题。声纹识别不需要精确的句子边界，只要一段足够干净的该说话人语音即可。

### 8.5 比对与判定

用 `matchSpeaker(embedding, library, 阈值)`：和库里每个人算余弦，取最高分，超过阈值（当前 0.31）判定为那个人，否则标“未识别”。当前代码还会在控制台打印每句的最高余弦分和最接近的人，用于调试阈值。

---

## 9. 隐私设计

- 声纹向量**只存浏览器本地 IndexedDB**，不上传任何服务器。
- 模型在浏览器里推理，音频不离开设备。
- 声纹按账号隔离，用户可随时删除（删除即从 IndexedDB 彻底移除）。

这符合声纹管理页对用户的“端到端、数据不出设备、随时可删”承诺。

---

## 10. 已知可调项 / 后续优化方向

- **识别阈值 0.31**：偏松会把别人认成你，偏紧会频繁“未识别”。最佳值需用真实会议的余弦分数分布来定（已加诊断日志辅助）。
- **注册质量**：注册时若环境吵或混入他人声音，声纹本身就不准，建议安静环境、单人录制。
- **断线健壮性**：长会议中若转写 WebSocket 断开，已做优雅收尾（停止采集并提示），不会再因疯狂重发导致页面崩溃。
- **ScriptProcessorNode 弃用警告**：麦克风采集用的是旧版 Web Audio API，控制台会有弃用警告，不影响功能，后续可迁移到 AudioWorkletNode。

## 11. 当前识别现象与原因拆解

结合当前会议页实现，页面里出现的三种说话人状态不是同一种含义：

- **已识别为某人**：声纹提取和比对都已完成，且最高分超过阈值，同时明显领先第二名。
- **未识别**：声纹提取和比对都已完成，但最高分未达到阈值，系统明确判定“不是库里任何一个人”。
- **识别中 / 识别中...**：这是一个挂起态，不是最终判定态。它可能来自两类情况：
  - 刚收到 ASR 最终句子，页面先用占位 speaker 出气泡，等待异步声纹识别回填。
  - 声纹识别过程中提前退出，或比对结果处于模糊区间，最终没有收口为“某个人”或“未识别”。

这也是当前体验不稳定的根因之一：用户看到的并不都是“识别失败”，而是相当一部分句子停留在 pending 状态。

### 11.1 当前代码中的关键分支

会议页里，`identifySpeakerForRecord()` 在以下场景会直接退出：

- 声纹引擎还没准备好（`vpReady === false`）
- 从环形缓冲切出来的 PCM 太短
- embedding 提取失败
- embedding 为空

而这些分支当前**不会把 pending speaker 收口为最终态**，所以气泡可能长期停在“识别中...”。

### 11.2 短句问题为什么特别明显

当前实现里，句子切窗时长先由 ASR 的 `timestampEnd - timestampStart` 计算，再夹到 `1~8 秒`；但后面又要求 PCM 至少达到 `1.5 秒` 才继续识别。

这会导致一个明显冲突：

- 一部分短句会被允许切窗
- 但切出来以后又因为 `< 1.5 秒` 被直接放弃
- 最终它们不会落成“未识别”，而是继续停在“识别中...”

因此，像“接一下”“嗯”“对”“可以”这类真实会议中的高频短句，会比长句更容易卡在 pending。

### 11.3 模糊句为什么会很多

当前实现除了阈值判定外，还额外做了一层“模糊区间”判断：

- 最高分达到阈值，不代表立即认人
- 如果最高分与第二名的差值小于 `0.05`
- 系统认为“分不清是谁”，会降级为“识别中”

这个策略的原意是减少误判，但在真实会议里，短句、压缩音频、系统声、多人音色相近等场景会显著增加“第一名和第二名差距很小”的概率，导致模糊句偏多。

## 12. 短句与模糊句最小调优方案

本节只针对当前最明显的两类问题做最小改动设计：

- 短句大量停留在“识别中...”
- 模糊句大量停留在“识别中”

本轮调优明确遵守以下边界：

- 只改声纹识别收口逻辑
- 只改 PCM 取窗、pending 降级、模糊区间
- 不改会议主流程
- 不改界面风格
- 不扩散到其他模块

### 12.1 调优目标

- **第一优先**：不要让 pending 永远挂着。
- **第二优先**：解决短句策略冲突。
- **第三优先**：减少模糊句长期停在“识别中”。

### 12.2 P0 必做项

#### P0-1. pending 必须收口

目标：

- 任何一句话都不能长期停在“识别中...”。

改动原则：

- `identifySpeakerForRecord()` 中所有提前退出的分支，不再直接结束。
- 这些分支统一进入一个“降级出口”。

建议降级顺序：

1. 优先沿用上一条已确认说话人的身份。
2. 如果不能安全沿用，则落成“未识别”。

适用场景：

- `vpReady` 未完成
- PCM 太短
- embedding 提取失败
- embedding 为空

产品收益：

- “识别中...” 只作为瞬时占位态，不再成为用户长期看到的最终状态。

#### P0-2. 统一短句 PCM 取窗规则

目标：

- 去掉当前 `1 秒切窗下限` 与 `1.5 秒最短识别长度` 的口径冲突。

建议方案：

- 将句子取窗最小值直接统一到 `1.5 秒`。
- 即：
  - 句长自适应逻辑仍保留
  - 但最小窗口不再是 `1 秒`
  - 而是 `1.5 秒`

建议口径：

```js
durationMs = Math.max(1500, Math.min(8000, durationMs));
```

产品收益：

- 减少“先切出来、后又因太短被放弃”的自相矛盾情况。
- 提升短句进入有效声纹提取流程的稳定性。

#### P0-3. 短句优先沿用上一说话人

目标：

- 提升真实会议中连续短句的识别体验。

适用句型：

- “接一下”
- “对”
- “可以”
- “嗯”
- “继续”

建议规则：

- 当当前句因 PCM 太短或提取失败无法可靠识别时：
  - 若上一条气泡是已确认说话人
  - 且不是“识别中 / 未识别”
  - 则短句优先沿用上一说话人
  - 否则落为“未识别”

限制条件：

- 只沿用最近一条
- 只用于短句或弱失败句
- 不允许无条件继承

产品收益：

- 更符合真实会议连续发言场景
- 比一律显示“识别中...”更自然
- 比一律打成“未识别”更符合用户预期

### 12.3 P1 次优先项

#### P1-1. 缩小模糊区间

目标：

- 降低“明明已经比较像，但仍然停在识别中”的比例。

当前参数：

- `VP_AMBIGUOUS_MARGIN = 0.05`

建议参数：

- 先保守调整到 `0.03`

原因：

- `0.05` 对真实会议场景偏保守
- 短句、远端压缩音频、音色相近的说话人会更容易被压进模糊区

注意：

- 这一步会略微增加误判风险
- 因此建议排在 pending 收口之后，而不是优先做

#### P1-2. 模糊句不再长期保留为“识别中”

目标：

- 将模糊句也收口为可解释的最终态。

建议规则：

- 对于最高分已过阈值、但与第二名差距不足的模糊句：
  1. 优先尝试沿用上一说话人
  2. 若不适合沿用，则落为“未识别”

不建议继续保留现状：

- `模糊 -> 识别中`

建议改成：

- `模糊 -> 沿用上一说话人`
- 或
- `模糊 -> 未识别`

产品收益：

- 让“识别中”回归真正的瞬时态语义
- 提升说话人标签的最终一致性

### 12.4 本轮建议暂不调整的项

为了保持最小修改面，本轮不建议同时调整以下内容：

- 主匹配阈值 `0.31`
- 识别前等待时间 `150ms`
- 注册页录音时长
- 注册页 embedding 生成逻辑
- 声纹库存储结构
- 会议主状态机

原因：

- 当前主要问题不是“整体都识别错”
- 而是“pending 不收口”和“短句/模糊句策略冲突”
- 若同时改太多参数，不利于判断每个改动的实际收益

## 13. 开发修改点清单

本清单用于后续开发逐项落地，按最小改动原则排列。

### 13.1 修改范围

本轮应尽量只改以下文件：

- `src/meeting/meeting-page-app.js`

原则上不需要改：

- `src/integrations/voiceprint-engine.js`
- `src/storage/voiceprint-store.js`
- `app/voiceprint.html`

### 13.2 需要调整的函数

#### 1. `identifySpeakerForRecord(record, result, channel)`

需要完成的改动：

- 为所有 early return 分支补统一的“最终收口”逻辑
- 将短句失败、引擎未就绪、embedding 失败、embedding 为空，统一接到降级处理
- 将模糊句从“继续识别中”改为“沿用上一说话人或未识别”
- 将句长最小切窗从 `1 秒` 调整为 `1.5 秒`

这是本轮最核心的修改点。

#### 2. 新增或抽出统一降级函数

建议抽出辅助函数，例如语义上类似：

- `resolveFallbackSpeaker(...)`
- `pickPreviousResolvedSpeaker(...)`
- `finalizePendingSpeaker(...)`

职责建议：

- 判断是否允许沿用上一说话人
- 若允许，返回上一位说话人
- 若不允许，返回“未识别”

这样可以避免把所有 fallback 逻辑都堆在 `identifySpeakerForRecord()` 里。

#### 3. 获取上一条有效说话人

建议新增一个只读辅助函数，语义上类似：

- 从 `segmentOrder` 倒序查找上一条已确认 speaker
- 排除以下 speaker：
  - `speaker-pending`
  - `speaker-unknown`

目标：

- 只让短句继承“明确已确认的人”
- 不继承 pending 或 unknown

### 13.3 具体开发动作

#### 动作 A：统一短句取窗

- 将最小切窗调整为 `1.5 秒`
- 保持最大值仍为 `8 秒`
- 避免后续再次因 `< 1.5 秒` 被立即放弃

#### 动作 B：补 pending 收口

- 所有失败出口不再直接 return
- 改为最终回填 speaker
- 确保页面上不留下长期“识别中...”

#### 动作 C：补短句继承策略

- 仅在短句 / 弱失败 / 模糊句中触发
- 仅沿用上一条已确认说话人
- 不对长句、明确低分句做无条件继承

#### 动作 D：缩小模糊区间

- `VP_AMBIGUOUS_MARGIN` 从 `0.05` 调整到 `0.03`
- 让一部分原本停在模糊区的句子进入明确判定

### 13.4 验收标准

完成后应重点验证以下场景：

- **短句场景**
  - “嗯”
  - “对”
  - “接一下”
  - “可以继续”
  - 预期：不再长期停在“识别中...”

- **连续发言场景**
  - 同一人连续说多句，其中夹杂短句
  - 预期：短句优先继承上一说话人

- **模糊句场景**
  - 两位声音相近的发言人连续说话
  - 预期：模糊句数量下降，不再长期保持 pending

- **明确低分场景**
  - 未录入声纹的说话人发言
  - 预期：落为“未识别”，而不是卡在“识别中”

### 13.5 本轮不应改变的产品口径

- 仍然保留“先出转写、后补说话人”的体验
- 仍然保留“未识别”作为标准失败态
- 不改变声纹注册入口和会议页布局
- 不改变当前声纹模型与本地存储方案
