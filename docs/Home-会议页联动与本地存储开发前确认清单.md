# Home-会议页联动与本地存储开发前确认清单

## 一、文档目标

本文档用于固定当前阶段“`home.html` 与 `index.html` 联动 + 会议本地存储 + 用户数据归属”的开发边界、实现方式、文件落点与分步骤实施顺序，作为后续逐步开发时的统一执行依据。

本清单只聚焦当前主线：

- `Home` 发起会议
- `Home` 返回进行中会议
- `Home` 渲染历史会议卡片
- `Index` 会中保存与会后归档
- 不同登录用户只看到自己的会议
- `Index` 后续支持历史回看模式

当前阶段不扩展到：

- `terms / voiceprint / admin` 的新业务联动
- 云端数据库
- 多人协同会议
- 多用户共看同一会议
- 搜索、筛选、收藏、导出等增强项

---

## 二、当前已完成基础

当前已具备以下基础条件：

- 已完成 `login.html` 的真实登录页
- 已完成 `admin / member` 两类 Demo 账号
- 已完成前端登录态存储与页面守卫
- 已完成 `settings.html` 中的退出登录能力
- 已完成 `app/index.html` 的真实会议页接入
- 已完成 `app/home.html` 的真实高保真主页壳

这意味着当前系统已经具备：

- “用户是谁”这一层
- “用户能否进入页面”这一层

当前还缺的是：

- `home` 上的会议卡还不是实时数据
- `index` 里的会议内容还没长期保存
- 不同用户登录后，还没有真正看到“只属于自己的会议”
- 历史回看还没形成真实数据流

本次开发前确认清单，就是为了解决这四个待办。

---

## 三、当前阶段主线结论

本阶段最重要的开发主线，不是先做复杂 UI，而是先做：

1. 会议存储模型
2. 用户归属规则
3. `home -> index` 页面接驳
4. 会后归档与历史读取

统一闭环定义为：

`当前登录用户 -> home 发起会议 -> index 实时会议 -> 会中持续保存 -> 会后归档 -> home 历史会议卡片 -> index 回看`

---

## 四、范围确认

### 4.1 本阶段必须实现

- `home.html` 的“开始新会议”接入真实逻辑
- `home.html` 的“回到会议”接入真实逻辑
- `home.html` 的历史会议区域改为真实数据渲染
- `index.html` 支持识别 `meetingId`
- `index.html` 在会中把会议内容持续写入本地
- `index.html` 在会议结束后写入最终归档结果
- 会议数据与当前登录用户建立归属关系
- 登录后仅能看到自己的会议列表
- 为后续 `replay` 模式预留并固定数据结构

### 4.2 本阶段明确不做

- 真后端数据库
- 真云端同步
- 真多人协作
- `admin / terms / voiceprint / settings` 的新功能扩展
- 历史会议搜索与筛选
- 收藏、标签编辑、删除会议
- 历史会议详情的新页面

## 五、总体设计原则

### 5.1 用户归属原则

一场会议默认由当前登录用户发起，并归属于该用户。

即：

- 谁点击“开始新会议”
- 会议就归谁

当前阶段不做：

- 一场会议归属于多个账号
- 多账号同时编辑一场会议

### 5.2 存储原则

会议数据先全部存本地，采用“总表 + `ownerUserId` 过滤”的方式实现多用户隔离。

采用该方案的原因：

- 更接近未来后端数据库结构
- 后续迁移云端最顺
- 便于统一查询进行中会议、历史会议与单场会议详情

当前阶段不采用“每个用户单独一个 storage key”的原因：

- 后续迁移成本更高
- 查询结构更分散
- 进行中会议与历史卡片的统一读取更麻烦

### 5.3 页面复用原则

`index.html` 继续作为唯一会议页面，后续通过参数区分：

- `live`
- `replay`

本阶段先优先把 `live` 模式与会后历史数据保存跑通，同时为 `replay` 模式固定数据结构和入口。

## 六、本地存储方案确认

### 6.1 本阶段使用的本地存储 key

继续保留当前已有：

- `ac_demo_users`
- `ac_current_user`

本阶段新增：

- `ac_meetings`

如有必要，可预留但暂不启用：

- `ac_meeting_drafts`

### 6.2 为什么先只做 `ac_meetings`

因为当前最重要的是把会议主闭环跑通：

- 创建
- 更新
- 结束
- 列表展示
- 历史读取

在 MVP 阶段，一条会议记录中同时保存“卡片摘要信息”和“完整会议内容”是可接受的。

如后续会议内容体积显著增大，再拆分：

- 会议索引
- 单场详情

当前阶段不必提前复杂化。

### 6.3 已确认的后续演进：会后本地后端备份

虽然当前阶段先采用浏览器运行时存储，但产品后续已确认将引入“本地后端写文件”的备份方案，作为会后历史会议的本地落盘能力。

该方案当前先记录，不在本阶段实现。

后续目标：

- 会议结束并生成最终总结后
- 前端继续正常写入浏览器本地存储
- 同时将该场会议数据 `POST` 给本地后端服务
- 由本地后端写入项目目录中的本地文件

建议写入位置：

- `data/meetings/`
- `data/exports/`

该方案的产品意义：

- 项目文件夹内可直接看到历史会议文件
- 更接近未来产品化时的服务端数据库 / 文件存储结构
- 在保留当前前端运行时流畅体验的同时，补上“本地可见归档”

当前约束：

- 本阶段仍以浏览器 `localStorage` 为主
- 本地后端备份不参与当前 `home -> index` 主链路开发
- 后续若开始实现本地后端，应新增独立文档或在本清单中继续扩写接口与目录规则

## 七、会议数据模型确认

### 7.1 总体模型

本阶段统一使用如下数据结构：

```json
{
  "id": "mtg_20260622_001",
  "ownerUserId": "u_member_001",
  "ownerName": "王敏",
  "title": "AI 会议平台产品介绍",
  "status": "ongoing",
  "startedAt": "2026-06-22T10:00:00.000Z",
  "endedAt": null,
  "durationSeconds": 0,
  "mainLanguage": "zh",
  "meetingTargetLanguage": "en",
  "participants": [],
  "tags": [],
  "transcriptSegments": [],
  "summary": null,
  "summaryStatus": "empty",
  "coverSnippet": "",
  "isStarred": false,
  "lastVisitedAt": "2026-06-22T10:00:00.000Z"
}
```

### 7.2 字段说明

- `id`：会议唯一标识，用于 `home -> index` 跳转
- `ownerUserId`：当前会议所属用户
- `ownerName`：用于调试与卡片展示
- `title`：当前阶段可先用默认标题，后续再支持编辑
- `status`：当前主要使用 `ongoing / finished`
- `startedAt / endedAt / durationSeconds`：用于进行中横幅和历史卡片
- `mainLanguage / meetingTargetLanguage`：用于回看时还原当前语言配置
- `participants`：从 transcript 中逐步整理，初版可允许为空
- `tags`：当前先保留字段，初版可以写死为空或默认值
- `transcriptSegments`：保存完整会中分段
- `summary`：保存最终结构化总结
- `summaryStatus`：`empty / ready`
- `coverSnippet`：历史卡片摘要文本
- `isStarred`：保留字段，当前阶段不重点开发
- `lastVisitedAt`：用于后续排序

### 7.3 transcript segment 结构确认

尽量复用当前会议页已经在内存中使用的数据结构：

```json
{
  "localSegmentId": "1-5-5",
  "serverSegmentId": 5,
  "speaker": {
    "name": "李明",
    "avatar": "李",
    "color": "var(--speaker-1)"
  },
  "sourceLanguage": "zh",
  "text": "我们先看这次产品演示的关键价值。",
  "translation": "Let's first look at the key value of this product demo.",
  "translationStatus": "success",
  "timestampEnd": 1710000000
}
```

### 7.4 summary 结构确认

直接保存当前 `meeting-summary` 模块已经使用的结构化结果：

```json
{
  "abstract": "本次会议主要围绕 AudioClaw 的跨语种会议能力展开。",
  "decisions": [],
  "actionItems": [],
  "openQuestions": [],
  "speakerStats": []
}
```

## 八、页面与流转确认

### 8.1 `home.html`

当前阶段承担职责：

- 登录后默认首页
- 渲染当前用户自己的会议
- 显示进行中会议横幅
- 显示历史会议卡片
- 发起新会议

### 8.2 `home -> index` 三种跳转

#### 开始新会议

点击位置：

- 左侧 `开始新会议`
- 欢迎区 `继续上次会议` 后续也可按同一逻辑接入

执行逻辑：

1. 读取当前登录用户
2. 创建一条会议记录
3. 默认状态为 `ongoing`
4. 跳转 `index.html?meetingId=xxx&mode=live`

#### 回到进行中会议

点击位置：

- `home` 页的“回到会议”横幅

执行逻辑：

1. 读取当前登录用户
2. 查询该用户 `status === ongoing` 的会议
3. 跳转 `index.html?meetingId=xxx&mode=live`

#### 历史会议卡片

点击位置：

- 历史会议卡片

执行逻辑：

1. 读取卡片对应的 `meetingId`
2. 跳转 `index.html?meetingId=xxx&mode=replay`

### 8.3 `index.html`

当前阶段承担职责：

- 作为实时会议工作区
- 作为后续历史回看页
- 根据 URL 参数读取 `meetingId` 与 `mode`

#### `live` 模式

- 允许点击开始会议
- 允许点击结束会议
- 连接 WebSocket
- 启动音频采集
- 产生并渲染实时内容
- 持续保存会议记录

#### `replay` 模式

本阶段先预留结构和入口，不必一步做到完整体验。

目标定义：

- 不连 WebSocket
- 不启动音频采集
- 从本地读取 `meetingId` 对应记录
- 渲染已保存的 transcript 与 summary

## 十一、详细开发顺序确认

### 步骤一：先落 `meeting-store.js`

目标：

- 固定会议模型
- 固定本地存储读写入口

原因：

- 所有后续页面联动都依赖它
- 先有存储层，页面逻辑才不会分散乱写 `localStorage`

### 步骤二：打通 `home` 的“开始新会议”

目标：

- 用户点击按钮后能创建一条属于自己的会议记录
- 跳转到带 `meetingId` 的 `index`

完成标准：

- 不同账号点击开始新会议，会生成不同归属的会议记录

### 步骤三：打通 `home` 的“回到会议”

目标：

- 如果当前用户已有进行中会议，就展示进行中横幅
- 点击后回到对应会议

完成标准：

- 李明只能回到李明的会议
- 王敏只能回到王敏的会议

### 步骤四：在会中持续保存

目标：

- 每次收到新 transcript segment 后更新本地会议记录
- 每次总结生成成功后更新本地会议记录

完成标准：

- 刷新或返回主页后，会议仍然保留

### 步骤五：会后归档

目标：

- 结束会议后写入最终 transcript、summary、结束时间和时长
- 状态改为 `finished`

完成标准：

- `home` 中可看到真实历史会议卡片

### 步骤六：历史会议列表渲染

目标：

- 用真实本地数据替换静态历史卡片

完成标准：

- 当前登录用户只看到自己的历史会议

### 步骤七：补 `replay` 模式

目标：

- 点击历史会议卡片后能进入只读回看

完成标准：

- 已保存的 transcript 与 summary 能在 `index` 中重新展示

## 十二、会中保存策略确认

当前阶段采用“关键节点立即保存”的方式，不先做复杂节流。

保存节点：

1. 创建会议后立即写入初始记录
2. 收到新的 segment 后写入
3. 翻译回填完成后写入
4. 总结生成成功后写入
5. 点击结束会议后写入最终归档

这样设计的原因：

- 简单直接
- 易于调试
- 对 MVP 足够可靠

## 十三、Home 页面展示规则确认

### 13.1 进行中会议

展示规则：

- 当前用户有且仅有一条 `ongoing` 会议时，展示进行中横幅
- 没有进行中会议时，进行中横幅隐藏

### 13.2 历史会议

展示规则：

- 当前用户的 `finished` 会议渲染为历史会议卡片
- 默认按 `lastVisitedAt` 或 `endedAt` 倒序排列

### 13.3 空态

展示规则：

- 当前用户没有任何会议时，显示空态提示
- 当前用户没有历史会议但有进行中会议时，只显示进行中横幅

---

## 十四、Index 页面状态规则确认

### 14.1 无 `meetingId`

当前阶段建议：

- `index.html` 在无 `meetingId` 情况下保留兼容，但不作为主要路径
- 正常流程应从 `home` 创建会议后进入

### 14.2 有 `meetingId` 且 `mode=live`

规则：

- 加载对应会议记录
- 允许实时会议逻辑运行

### 14.3 有 `meetingId` 且 `mode=replay`

规则：

- 禁止启动实时采集
- 禁止触发真实开始/结束逻辑
- 直接读取本地历史数据

## 十六、风险提醒

### 16.1 `localStorage` 容量风险

会议 transcript 长度增长后，单条会议记录可能变大。

当前策略：

- 先保证 MVP 跑通
- 后续若数据量明显增大，再评估 `IndexedDB`

### 16.2 状态污染风险

当前 `meeting-page-app.js` 以单页内存状态为主，接本地持久化时要注意：

- 新会议与旧会议不能混用同一份运行状态
- `replay` 模式不能误启动 `live` 流程

### 16.3 页面静态内容替换风险

`home.html` 目前有大量静态卡片样例，真实接入时要注意：

- 先替换主链路核心区域
- 不要一次性重做整个页面
- 优先保证“开始新会议 / 回到会议 / 历史会议卡片”可用

---

## 十七、最终确认

本阶段的实际开发顺序，确认如下：

1. 先做 `meeting-store.js`
2. 再做 `home -> 开始新会议`
3. 再做 `home -> 回到会议`
4. 再做会中持续保存
5. 再做会后归档
6. 再做历史会议列表
7. 最后补 `replay` 模式

后续中途若有实现细节调整，应以本清单为准更新，而不是直接在代码层无文档变更地漂移实现方向。
