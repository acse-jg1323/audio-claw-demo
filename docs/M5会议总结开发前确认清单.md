# M5会议总结开发前确认清单

## 一、文档目的

这份清单用于在正式开发 `M5. 会议总结` 前，快速确认还需要拍板的产品细节、接口字段、按钮位置和编辑逻辑。

功能方向、工作流与 Summary Agent 方案已在：

- [AI会议总结功能开发设计.md](file:///Users/gejiaqi/Desktop/需求与原型\(1\)/AI会议总结功能开发设计.md)

本清单只保留“开发前必须确认”的关键点。

***

## 二、当前已确认内容

- `M5` 优先于 `M4` 开发
- `M5` 与实时转写 `segment` 强绑定
- 支持两种触发方式：
  - `会中手动触发`
  - `会后自动触发`
- 整体流程固定为：
  - `segment 收集 -> 发言占比计算 -> Summary LLM -> JSON 输出 -> 前端渲染 -> 手动编辑 -> 导出`
- `参会人发言占比` 由程序计算
- 其余 4 个模块由 Summary Agent 一次结构化输出
- 总结页第一版优先做 `单语言总结`
- 导出第一优先级是 `Markdown`

***

## 三、开发前待确认项

## 3.1 页面按钮与位置：本部分都确认为“是”

### 已有原型位置

当前总结页结构位于：

- [index.html](file:///Users/gejiaqi/Desktop/需求与原型\(1\)/index.html#L1464-L1543)

当前原型顶部已有：

- 标题
- `导出` 按钮

### 需要确认

- 是否在总结页顶部新增一个主按钮：
  - `生成当前总结`
- 当会议已结束时，该按钮是否切换为：
  - `生成最终总结`
- 生成完成后，顶部主按钮是否切换为：
  - `重新生成`
- `编辑总结` 按钮是否放在顶部标题区，与导出并列
- `保存修改 / 取消编辑` 是否在进入编辑态后替换顶部按钮

### 当前推荐

- 标题区右侧放 2 个主操作：
  - `编辑总结`
  - `导出`
- 在空态或未生成时，把 `生成当前总结 / 生成最终总结` 放在内容区主视觉位置
- 生成完成后，如果需要 `重新生成`，建议仍放在标题区右侧，优先级低于 `编辑总结`

***

## 3.2 总结页状态展示

### 需要确认

总结页第一版是否按以下状态做：

- `empty`
- `loading`
- `ready`
- `editing`
- `error`

### 当前推荐

- 采用以上 5 种状态
- 不再保留只有“静态默认内容”这一种状态

### 需要拍板的表现细节

- `empty` 状态是否显示：应当以适当的方式占位展示
  - 一段说明文案
  - 一个主按钮
- `loading` 状态是否使用 skeleton，而不是简单文字 `生成中 ：如果指的是loading有一个过程骨架，是的，比简单文字要好`
- `error` 状态是否提供：应当提供
  - 错误文案
  - 重试按钮

***

## 3.3 总结输出语言

### 已确认方向

- 第一版优先做 `单语言总结：没问题`

### 还需确认

- `summary_language` 是否默认跟随：
  - `主阅读语言：是的`
- 是否允许用户在总结页里临时切换总结输出语言：应当允许，但暂时先只做跟随主阅读语言的吧！注意后续输入给大模型返回结果可以指定语言这样比较好
- 如果允许切换，切换语言时是否：
  - 重新生成总结
  - 还是只做界面层切换

### 当前推荐

- `summary_language` 默认跟随 `主阅读语言`
- 第一版**不在总结页单独提供语言切换按钮**
- 若未来要支持多语言总结，后续再补，不放进 M5 第一轮

***

## 3.4 Summary Agent 输入字段

### 当前设计中建议输入

- `meeting_id`
- `meeting_title`
- `summary_mode`
- `summary_language`
- `main_language`
- `meeting_target_language`
- `participants`
- `segments`
- `speaker_stats`

### 开发前还需确认

- `segments` 是否携带以下字段就够：
  - `segment_id`
  - `speaker`
  - `speaker_role`
  - `timestamp_end`
  - `source_language`
  - `text`
  - `translated_text`
- 是否要把 `server session_id` 一类调试字段排除在 Summary 输入之外：对summary没用只会增加上下文长度的，就可以去掉，尽量让大模型不接受过多冗余信息
- `participants` 是否使用当前页面里展示名即可：对，目前界面上这几个名字就是我们暂时先用来做计算的，没问题

### 当前推荐：没问题

- Summary 输入中不带无关调试字段
- `participants` 直接使用页面展示名
- `segments` 保持轻量，不加多余原始响应字段

***

## 3.5 Summary Agent 输出字段

### 当前设计中建议输出

- `summary_version`
- `summary_mode`
- `summary_language`
- `title`
- `ai_summary`
- `decisions`
- `action_items`
- `open_questions`
- `metadata`

### 还需确认

- `title` 是否由 Agent 生成，还是由前端固定生成
- `metadata.confidence_note` 是否保留在第一版
- `action_items.owner_status` 是否保留

### 当前推荐

- `title` 先由前端固定拼装，不交给模型
- `metadata.confidence_note` 第一版可保留，但前端可不展示
- `owner_status` 建议保留，方便后续区分：
  - `confirmed`
  - `unclear`

***

## 3.6 原文还是译文参与总结

### 已确认方向：没问题

- 不建议只丢原文
- 也不建议只丢译文

### 还需确认

- M5 第一版是否采用：
  - `summary_language 统一后的 summary_segments：是的`
  - 作为唯一送入 Agent 的正文
- 原始 `text` 是否只作为后续保留字段，不直接进第一版 prompt

### 当前推荐

- 第一版把 `summary_segments` 作为 Agent 主要正文输入：是的
- 原始 `text` 暂不直接塞进 prompt
- 这样更稳、更省 token，也更适合单语言总结输出：没错

***

## 3.7 发言占比计算口径

### 已确认方向

- 发言占比由程序计算

### 还需确认

- 第一版按什么口径算：
  - 字符数
  - segment 数
  - 时长

### 当前推荐：没问题，可解释性强实现简单，OKOK

- 第一版按 `字符数占比`
- 原因：
  - 实现最简单
  - 可解释性强
  - 不依赖真实说话人时长能力

***

## 3.8 编辑态细节

### 已确认方向

- AI 结果应支持手动编辑

### 还需确认

- `AI 摘要` 是否用 `textarea`
- `关键决议` 是否支持：
  - 增加一条
  - 删除一条
- `待办与负责人` 是否支持编辑：
  - 任务
  - 负责人
  - 截止日期
- `遗留问题` 是否支持增删
- `发言占比` 是否只读

### 当前推荐

- `AI 摘要`：`textarea`
- `关键决议`：可增删列表
- `待办与负责人`：结构化可编辑项
- `遗留问题`：可增删列表
- `发言占比`：只读

***

## 3.9 导出范围与顺序

### 已确认方向

- 邮件正文暂不做
- 优先做前三个导出方向

### 还需确认

- 第一阶段是否只先做：
  - `Markdown`
- `PDF / Word` 是否延后到 M5-6 第二轮

### 当前推荐

- `M5-6` 第一轮先只做 `Markdown`
- `PDF / Word` 放到第二轮

原因：

- 更快闭环
- 更适合先验证总结结构和编辑结果

***

## 3.10 前后端接口边界

### 还需确认

- M5 第一轮是否：
  - 仍允许前端直连模型做验证
  - 还是直接切到后端 Summary 接口

### 当前推荐

- 若目标是快速验证总结效果：
  - 第一轮可以前端直连模型验证 Summary JSON 输出
- 若目标是正式进入产品开发：
  - 尽快收口到后端接口

对于后续正式产品，推荐接口如下：

- `POST /api/meeting-summary/generate`
- `POST /api/meeting-summary/save`
- `GET /api/meeting-summary/:meeting_id`
- `POST /api/meeting-summary/export/markdown`

## 四、当前建议的默认拍板版本

如果你希望尽快进入 M5 开发，当前推荐直接默认采用下面这一版：

- 总结页支持 `empty / loading / ready / editing / error`
- 顶部保留 `导出`
- 内容区空态主按钮使用：
  - `生成当前总结`
  - 会议结束后切换为 `生成最终总结`
- `编辑总结` 放在标题区右侧
- 总结语言默认跟随 `主阅读语言`
- 第一版不做总结语言切换按钮
- Agent 输入正文使用 `summary_segments`
- 发言占比按字符数计算
- `AI 摘要 / 关键决议 / 待办与负责人 / 遗留问题` 支持手动编辑
- `发言占比` 只读
- 导出第一轮先做 `Markdown`
- M5 第一轮可先直连模型验证，再收口后端
