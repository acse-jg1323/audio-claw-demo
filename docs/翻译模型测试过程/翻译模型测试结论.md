# 翻译模型测试结论

> 测试时间：2026-06-18
> 测试范围：`senseaudio-s2-flash`、`senseaudio-s2-lite`、`senseaudio-s2`
> 测试方式：统一 prompt、统一 9 条样本、两轮本地实测

## 一、结论摘要

本轮测试后，当前最适合作为智能会议系统 MVP“识别后路由翻译”执行器的模型是：

- `senseaudio-s2-lite`

推荐理由：

- 延迟最低且最稳定
- 输出格式稳定，基本都只返回译文
- 中日英互译都可用
- API、WebSocket、token、AudioClaw 等术语保留表现可接受
- 更适合会中逐 segment 的近实时翻译场景

当前不建议作为 MVP 默认翻译执行器的模型：

- `senseaudio-s2-flash`

原因：

- 虽然部分样本响应很快
- 但两轮实测都出现了明显长尾延迟
- 长尾达到十几秒到数十秒，不适合会中实时体验

可以作为质量参考但不适合作为会中默认链路的模型：

- `senseaudio-s2`

原因：

- 翻译质量整体稳定
- 术语保留较好
- 但平均延迟显著高于 `senseaudio-s2-lite`
- 更适合后续做高质量模式或会后重翻参考

## 二、两轮测试平均延迟

### 第一轮

- `senseaudio-s2-flash`：约 `11098 ms`
- `senseaudio-s2-lite`：约 `268 ms`
- `senseaudio-s2`：约 `4089 ms`

### 第二轮

- `senseaudio-s2-flash`：约 `14295 ms`
- `senseaudio-s2-lite`：约 `279 ms`
- `senseaudio-s2`：约 `3087 ms`

## 三、模型表现判断

### 1. senseaudio-s2-lite

判断：

- 当前最适合 MVP 会中翻译

优点：

- 延迟稳定在几百毫秒量级
- 中译日、日译中、英译中表现都够用
- 对技术术语、英文产品名、夹杂词场景适配较好
- 输出基本不带解释性噪音

风险：

- 某些句子略偏直译
- 日文风格不如旗舰模型圆润

总体评价：

- `推荐作为 MVP 默认翻译执行器`

### 2. senseaudio-s2

判断：

- 质量稳定，但速度偏慢

优点：

- 翻译更完整
- 中文口语与技术句的表达更稳
- 更像高质量参考版本

风险：

- 3 秒级延迟对于会中逐 segment 翻译偏慢
- 不适合作为默认链路

总体评价：

- `推荐作为质量上限参考 / 会后重翻备选`

### 3. senseaudio-s2-flash

判断：

- 当前不适合作为会中默认翻译模型

优点：

- 个别样本响应很快
- 文本质量本身不差

风险：

- 长尾延迟明显
- 两轮测试都出现十几秒甚至二十多秒延迟
- 稳定性不符合会中实时产品要求

总体评价：

- `当前不推荐用于 MVP 默认链路`

## 四、对产品落地的直接建议

当前建议的翻译链路为：

- `DeepThink WebSocket` 负责实时原文转写
- 规则语言判别模块负责产出 `source_language`
- 翻译路由层负责产出 `target_language`
- `senseaudio-s2-lite` 负责逐 segment 文本翻译
- 页面先展示原文，再异步补写译文

## 五、当前阶段建议

- MVP 默认翻译模型：`senseaudio-s2-lite`
- 备选模型：`senseaudio-s2`
- 当前不建议默认使用：`senseaudio-s2-flash`

## 六、后续可继续补测的方向

- 加入术语表约束后的表现
- 加入 `context_tail` 与不加 `context_tail` 的对比
- 更长口语句和更短碎句的对比
- 更高频调用下的吞吐稳定性
