# ASR接入理解与前端落地方案

> 版本：v0.1
> 目标：明确当前智能会议 MVP 在 ASR 能力接入上的技术判断、前端落地方式与当前阶段开发边界。

## 一、当前结论

基于现有官方文档、模型能力表和本地测试页，可以先得出 5 个明确结论：

- 当前会议 MVP 的实时主链路，应以 `senseaudio-asr-deepthink-1.5-260319` 为核心模型
- 当前官方 WebSocket 实时接口只支持 `DeepThink`，不适合一开始做多模型兼容
- 同事的本地测试页主要验证了 **HTTP 文件识别接口** 已跑通，对模型能力理解有帮助，但不能直接作为实时会议页接法
- 当前 MVP 的第一阶段重点不是做全量产品，而是先打通 `实时识别链路`
- 当前阶段只做两步：`ASR接口验证` 和 `会议工作台改造`

## 二、四个模型对当前项目的意义

### 1. Lite

特点：

- 支持基础识别
- 支持多语言
- 支持热词增强
- 不支持流式返回
- 不支持翻译

判断：

- 适合轻量、低成本、偏离线的基础识别场景
- 不适合作为当前会议系统的实时主模型

### 2. Standard

特点：

- 支持基础识别
- 支持多语言
- 支持字级 / 句级时间戳
- 不支持流式返回
- 不支持翻译

判断：

- 适合离线转写、时间轴整理
- 不适合作为当前会议系统的实时主模型

### 3. Pro

特点：

- 支持基础识别
- 支持说话人分离
- 支持字级 / 句级时间戳
- 支持翻译
- 不支持流式返回

判断：

- 非常适合会后精修、历史会议整理、离线会议回看
- 不适合当前 MVP 的会中实时页主链路

### 4. DeepThink

特点：

- 支持基础识别
- 支持流式返回
- 支持翻译
- 支持智能编辑收敛
- 不支持说话人分离
- 不支持精细时间戳能力

判断：

- 它是当前实时会议 MVP 的核心模型
- 当前的实时会议页、实时字幕、实时翻译都应优先围绕它设计

## 三、本地测试页的价值与边界

工作区中的本地测试页：

- `senseaudio-test.html`

它的价值：

- 证明同事已经把 HTTP 文件识别接口跑通
- 证明四个模型的离线识别 / 翻译效果已经做过对比验证
- 证明你们当前 API key 在开发阶段具备可测性

它的边界：

- 它走的是 `https://api.senseaudio.cn/v1/audio/transcriptions`
- 它是 `fetch + multipart/form-data + 文件上传`
- 它不是 WebSocket 实时识别
- 它没有覆盖会议页真正需要的“边录边传、边识别边展示”链路

因此，这个页面对我们有帮助，但它更像：

- `离线模型验证页`

而不是：

- `实时会议接入页`

## 四、当前会议 MVP 应该怎么接

当前会议 MVP 的 ASR 主链路建议如下：

- 模型：`senseaudio-asr-deepthink-1.5-260319`
- 协议：`WebSocket`
- 接口：`wss://api.senseaudio.cn/ws/v1/audio/transcriptions`
- 音频格式：`PCM / s16le / 16kHz / 单声道`
- 交互模式：前端持续上传二进制音频帧，服务端持续返回分段识别结果

也就是说，会议页不应继续沿用“上传文件再返回结果”的思路，而应该改成：

`开始会议 -> 建立WS连接 -> 发送task_start -> 推送实时音频帧 -> 接收result_final -> 页面渲染 -> task_finish`

## 五、鉴权理解

当前资料里存在一个需要明确的点：

- 一份说明中写的是：WebSocket 接口必须通过服务端先申请临时 Token，再由前端使用 Token 鉴权
- WebSocket 文档页中又写了：可以通过 `Authorization: Bearer ...` 直接传入鉴权信息

基于产品落地和安全性考虑，当前建议采用更稳妥的方案：

### 开发期建议

- 本机联调阶段，可以临时验证直接鉴权是否可用
- 但不把长期密钥写进前端代码

### 正式实现建议

- 后端安全保存长期密钥 / Public Key
- 后端提供一个“获取临时 Token”的轻量接口
- 前端每次进入会议页时先向后端请求临时 Token
- 前端拿临时 Token 建立 WebSocket 连接

最终判断：

- `正式产品实现必须有一个轻量后端鉴权层`
- 即使整个系统后端很轻，也不能完全没有这个层

## 六、WebSocket 实时识别的标准流程

### 第一步：建立连接

前端连接：

- `wss://api.senseaudio.cn/ws/v1/audio/transcriptions`

连接成功后，服务端返回：

- `connected_success`

### 第二步：发送 task\_start

前端发送 JSON 控制消息，至少包括：

- `event`
- `model`
- `audio_setting`
- `vad_setting`

建议第一版先用最保守配置：

```json
{
  "event": "task_start",
  "model": "senseaudio-asr-deepthink-1.5-260319",
  "audio_setting": {
    "sample_rate": 16000,
    "format": "pcm",
    "channel": 1
  },
  "vad_setting": {
    "silence_duration": 500,
    "min_speech_duration": 300
  }
}
```

服务端返回：

- `task_started`

### 第三步：发送音频二进制流

前端将实时采集到的 PCM 音频按分片发送给服务端。

建议分片：

- 每片约 `100ms`
- 约 `3200 字节`

### 第四步：接收识别结果

服务端按 VAD 断句返回：

- `result_final`

当前前端重点使用的字段：

- `data.text`
- `data.segment_id`
- `data.timestamp_end`
- `data.translations`（如果返回）

### 第五步：结束任务

会议结束时，前端发送：

```json
{ "event": "task_finish" }
```

服务端返回：

- `task_finished`

## 七、前端应该如何理解返回结果

对当前会议 MVP 来说，前端不要把 SenseAudio 返回当作“复杂 AI 对象”，而应把它理解成一条条会议消息。

建议前端内部先统一成如下结构：

```js
{
  id: "segment-1",
  segmentId: 1,
  text: "原始识别文本",
  translation: "翻译文本",
  timestampEnd: 1773027072669,
  isFinal: true,
  source: "mic",
  speaker: null
}
```

这里要特别注意：

- 当前 DeepThink 方案不应假设一定有说话人分离
- 当前阶段也不要依赖精细字级时间戳
- 第一版只需要稳定渲染“分段文本 + 可选翻译”

## 八、对 `index.html` 的直接影响

当前 `原型/index.html` 是静态原型，下一步要把它改造成真正可接 ASR 实时流的会议页。

第一版只建议改这几个区域：

### 1. 顶栏状态区

需要接入真实状态：

- 未开始
- 连接中
- 转写中
- 已结束
- 出错

### 2. 实时转写区

需要替换成真正的数据渲染容器：

- 能动态插入一条条识别结果
- 能展示原文
- 能展示翻译
- 能自动滚动到底部

### 3. 开始 / 结束会议行为

当前静态按钮要改成真实动作：

- 开始会议：请求 token -> 建立连接 -> 发送 task\_start -> 开始推流
- 结束会议：停止采集 -> 发送 task\_finish -> 等待 task\_finished

### 4. 总结 Tab

当前阶段先保留静态结构，不在第一步就接入真实总结能力。

原因：

- 现在的第一目标是打通实时链路
- 会后总结属于下一阶段

## 九、当前阶段只做的两步

基于当前项目节奏，建议严格收敛范围，只做以下两步。

## 第一步：ASR 接口验证

目标：

- 跑通 DeepThink WebSocket
- 明确前端采音方式
- 明确消息格式
- 明确识别结果结构

产出：

- 一个最小 WebSocket 联调 demo
- 一份最终确认后的字段说明

## 第二步：会议工作台改造

目标：

- 把 `原型/index.html` 改造成真实会议页
- 接入实时文本展示
- 跑通开始会议与结束会议的完整链路

产出：

- 一版可运行的实时会议页

## 十、当前阶段明确不做

为了让开发节奏稳定，当前阶段不建议一起做这些内容：

- 多模型切换
- 复杂说话人分离
- 精细时间戳播放器
- 历史会议完整回看
- 总结自动生成
- 术语库联动
- 声纹库联动
- 管理后台联动

这些都属于后续增强，而不是当前第一步和第二步的目标。

## 十一、下一步需要你继续提供的内容

为了正式进入联调，接下来最有价值的是继续明确以下信息：

- WebSocket 是否支持直接使用长期密钥，还是必须先走临时 Token，如果走临时 Token，后端准备如何申请和下发（先走长期密钥）
- 当前你希望前端优先采集哪一路音频，是先采集麦克风，还是系统音频（优先采用麦克风）
- 是否需要第一版就展示翻译（是的，同时展示原始识别语言和翻译是重要功能，需要第一版就出。应当支持点选原始语言和翻译语言。如果不设置，默认原始语言为中文，默认翻译语言为日文。）
- 是否已经有总结接口或会后处理接口（没有，会议中内容开发完后这个是下一步最急着要做的）
- 补充：当前第一步到第二步的开发不要在原始原型上开发，原型/.仅作为对比参考。要在新的可用的工作区以良好的模块化软件开发思路去搭建程序。

## 十二、一句话结论

当前智能会议 MVP 的 ASR 接入路线已经可以明确为：

`用 DeepThink 跑通 WebSocket 实时识别 -> 把 index.html 改造成真实会议工作台 -> 先稳定跑会中链路，再做后续增强`
