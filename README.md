# AudioClaw 智能会议系统 MVP

## 项目简介

这是一个以浏览器为载体的 AI 智能会议系统 MVP 项目，当前重点验证以下主链路：

- 登录与最小权限
- 会中实时转写与翻译展示
- 会后 AI 总结
- `home -> 新建会议 -> live -> 归档 -> 历史卡片 -> replay` 会议闭环

当前项目以静态页面 + 前端模块为主，真实页面位于 `app/`，高保真原型保留在 `原型/` 作为视觉与交互参考，不作为直接开发入口。

## 目录说明

- `app/`：当前真实可运行页面入口
- `src/`：前端业务逻辑与模块代码
- `docs/`：需求、方案、开发前确认清单、阶段看板等文档
- `原型/`：高保真原型留档，仅作为参考
- `scripts/`：独立脚本与测试脚本

## 本地启动

本项目当前采用静态服务运行。

### 1. 启动服务

在项目根目录执行：

```bash
python3 -m http.server 8000
```

### 2. 访问地址

优先从登录页进入：

```text
http://localhost:8000/app/login.html
```

也可按需直接访问：

- `http://localhost:8000/app/home.html`
- `http://localhost:8000/app/index.html`

## 当前运行说明

- 当前会议历史与登录态主要保存在浏览器 `localStorage`
- 当前产品的效果验证在服务器为Mac系统、Chrome浏览器上。若Win系统样式有不同，暂时以视觉效果看得过去就不改动为准。
- 当前历史会议存储 key 以前端运行时数据为主，不会自动写入项目目录；后续希望能加入本地会议存储（非数据库，以总会议文件夹存储json的方式实现）
- 不同电脑、不同浏览器之间不会自动共享本地历史数据
- 后续如需把会议归档真正写入项目文件夹，需增加本地后端能力

## 协作开发约定

以下约定来自当前项目已经形成的稳定开发口径，后续协作默认遵守。

### 1. 需求与文档回溯

- 所有原始需求优先回溯 `docs/系统功能与逻辑梳理.md`
- 所有 MVP 范围、阶段成果、已完成情况优先查看 `docs/MVP开发任务分解看板.md`
- 涉及某一主链路的大任务开发，必须先在 `docs/` 落一份方案或开发前确认清单，再进入代码实现
- 已有专项方案应优先延续，不应绕开既有文档另起一套口径

### 2. 原型一致性

- 一切开发默认以高保真原型为视觉与交互基准
- 非用户明确要求，不得擅自改动页面风格、布局层级、文案气质和核心交互路径
- 功能接入时应尽量采用“挂载真实逻辑、不破坏既有样式”的方式实现
- 若必须调整样式，目标应是更贴近原型，而不是为了开发方便重做一版 UI

### 3. 变更原则

- 每次修改都应尽量最小化，只解决当前明确范围内的问题
- 不得因为修一个功能，顺手改动无关页面、无关结构或无关文案
- 涉及已有稳定功能时，优先做增量接入，不做推翻式重构
- 遇到目录调整、路径修改、页面接驳时，优先保证现有功能不回退

### 4. 大任务执行方式

- 在开发一个大任务前，先完成方案文档、开发前确认清单或落地说明
- 大任务应拆成小步骤逐步实施，并在每一步完成后再进入下一步
- 产品逻辑存在歧义时，先对齐口径，再开发
- 重要链路开发完成后，应同步更新阶段看板或相关方案文档

### 5. 页面与模块边界

- `app/` 中的是当前真实页面，开发与联调应以这里为准
- `原型/` 中的是留档原型，默认不作为运行入口，不随功能开发一起随意改动
- 新功能优先接入现有页面与模块，不轻易新增平行页面
- 公共逻辑优先放入 `src/` 模块中，不把大量业务逻辑直接堆在 HTML 里

### 6. MVP 优先级原则

- 当前以 MVP 可演示闭环为第一优先级
- 优先完成主链路，不优先做复杂后端、复杂权限、复杂协同
- 本阶段允许使用 `localStorage` 做运行时临时存储，以换取更快开发和更稳展示
- 已明确为后续规划的能力，应先记录文档，不强行提前实现

### 7. 会议闭环相关约定

- `home` 顶部横幅只承担“进行中会议 / 回到会议”作用
- 历史会议卡片区只展示已归档会议，不展示 `ongoing`
- 右上角“结束”只负责结束会中流程，不代表会议最终归档
- AI 总结完成后，通过“归档并返回首页”完成最终收束
- 历史卡片进入的是 `replay` 只读回看模式，不应再启动实时链路

### 8. 数据与存储约定

- 当前会议数据采用总表思路存储，并通过 `ownerUserId` 做用户隔离
- 本地存储当前用于 MVP 展示，不等价于正式持久化方案
- 若后续增加删除、筛选、导出等历史管理能力，应基于现有会议存储模型增量扩展
- 若后续实现本地文件落盘，应通过本地后端写入 `data/` 目录，而不是直接由前端写工程文件

### 9. 质量与验证约定

- 每次改动后至少自查本功能链路是否可跑通
- 如果修改影响页面接驳、登录守卫、会议流转等关键路径，应补做一次端到端验证
- 文档、代码、页面实际行为三者应尽量保持一致
- 若发现当前实现与既有文档不一致，应先修正文档或先完成口径确认

### 10. 协作与提交建议

- 提交前优先整理本次改动范围，避免把无关改动混在一起
- 提交说明建议直接写清楚本次修改针对哪个阶段或哪条链路
- 如接手人需要快速理解项目，请先阅读：
  - `README.md`
  - `docs/系统功能与逻辑梳理.md`
  - `docs/MVP开发任务分解看板.md`
  - 对应专项方案文档

## 当前已知限制

- 会议历史默认只存在浏览器本地，不随仓库同步
- 若外部 ASR / WebSocket 配置缺失，页面可打开，但实时能力未必可完整运行
- 本项目当前仍以演示型 MVP 为主，部分增强能力尚未进入高优先级开发
