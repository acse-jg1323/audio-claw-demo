# 人员管理与声纹联动开发方案

## 1. 文档目的

这份文档只用于明确后续开发口径，不涉及本次代码修改。

目标是把以下三件事收敛成一套统一、可执行、可持续扩展的方案：

1. 把管理员后台的“成员管理”从高保真静态界面做成真实数据管理。
2. 把声纹库从“仅个人声纹资产”升级为“可与企业成员联动，也可支持外部来宾”的真实管理系统。
3. 为后续“会议中选择真实参会人员”“会议中用真实人员搜索”预留稳定的数据基础。

## 2. 本轮目标

本轮要实现的业务目标，不分先后，但必须统一设计：

1. 管理员可以新建、修改、删除企业成员。
2. 声纹库新增时支持两种方式：
   - 选择一个企业内部员工并自动带出姓名与职位
   - 自定义填写一个外部人员并录入声纹
3. 声纹记录支持下载。
4. 声纹管理页中的姓名、职位可修改。
5. 管理界面的筛选和搜索要真实可用。
6. 当前高保真界面中的三十多位成员要落成真实数据，但页面视觉与内容保持一致。
7. 后续会议页的“参会人员选择”要能够复用这套真实人员主数据。

## 3. 核心产品判断

### 3.1 人员系统与声纹系统不是两套“人”

后续系统里，“成员管理”和“声纹管理”不能再各自维护一套独立人名。

正确关系应当是：

- 管理后台维护的是“组织成员主档”。
- 声纹库维护的是“声纹资产”。
- 声纹资产可以绑定到某个组织成员。
- 声纹资产也可以不绑定成员，表示“外部来宾”。

也就是说：

- 成员是“人”的主数据。
- 声纹是“这个人对应的一种识别资产”。
- 会议中的说话人识别，后续优先应回填到“成员主档”而不是只回填一个纯文本姓名。

### 3.2 组织成员与系统登录账号不是一回事

当前系统已有登录用户概念，但后续“组织成员”不能直接等同于“登录账号”。

原因：

- 企业成员可能暂时未激活。
- 企业成员可能只是通讯录对象，还没有实际登录。
- 外部来宾可能永远不会成为系统账号。
- 一个管理员仍然需要维护整个组织的通讯录与声纹资产。

因此，后续数据层至少要区分：

- `账号用户`
- `组织成员`
- `声纹资产`

## 4. 产品范围

## 4.1 本轮纳入范围

- 管理后台成员真实数据化
- 成员新增 / 编辑 / 删除
- 成员筛选 / 搜索
- 声纹记录与组织成员绑定
- 声纹记录支持“内部成员”与“外部来宾”两类
- 声纹记录编辑姓名 / 职位
- 声纹记录下载
- 用真实数据替换当前高保真界面的静态成员列表

## 4.2 本轮不纳入范围

- 不做真正的后端组织系统
- 不做邮件真实发送
- 不做权限审批流
- 不做成员头像上传
- 不做复杂组织架构树
- 不做岗位字典配置
- 不做会议中多声纹合并、别名管理、多语言姓名映射

## 5. 统一数据关系

后续要建立以下关系：

```text
组织 Organization
  └─ 成员 Member
       ├─ 可被管理员维护
       ├─ 可被会议参会人选择
       └─ 可被声纹记录绑定

声纹 Voiceprint
  ├─ 可以绑定某个 Member
  └─ 也可以不绑定 Member，表示外部来宾 External

会议 Meeting
  └─ 说话人 Speaker
       ├─ 优先关联 Voiceprint
       ├─ Voiceprint 若已绑定 Member，则进一步关联 Member
       └─ 无绑定时保留文本兜底
```

## 6. 数据模型设计

## 6.1 组织成员主档

建议新增统一成员存储，例如：

- `src/storage/org-member-store.js`

建议字段如下：

```js
{
  id: "mem_xxx",
  companyKey: "神泉科技",
  name: "田中 健太",
  email: "kenta@company.com",
  role: "owner | admin | member",
  department: "销售",
  title: "",
  status: "active | pending | disabled",
  lastActiveAt: "2026-07-05T10:00:00.000Z",
  joinedAt: "2026-06-01T09:00:00.000Z",
  avatarText: "田",
  isSeeded: true,
  createdAt: "...",
  updatedAt: "..."
}
```

字段口径说明：

- `id`：组织成员稳定主键，后续会议、声纹、搜索都依赖它，不能再只靠姓名。
- `companyKey`：组织作用域。
- `role`：只允许 `owner/admin/member`。
- `department`：与管理界面筛选项一致。
- `title`：成员职位，管理表格仍不展示，但新增/编辑弹窗必须支持录入。
- `status`：用于支撑“所有者 / 管理员 / 普通成员 / 待激活 / 已停用”等筛选维度。
- `avatarText`：继续兼容现有高保真 UI 的姓名首字展示。

## 6.2 声纹资产

现有声纹存储建议继续沿用，但要补充“成员绑定”能力。

建议扩展字段为：

```js
{
  id: "vp_xxx",
  ownerUserId: "u_admin",
  memberId: "mem_xxx" || null,
  personType: "internal | external",
  companyKey: "神泉科技",
  name: "田中 健太",
  role: "CTO",
  color: "var(--accent-blue)",
  avatar: "田",
  embedding: [...],
  sampleCount: 123,
  createdAt: "...",
  updatedAt: "..."
}
```

字段口径说明：

- `memberId`：
  - 内部成员声纹：必须绑定
  - 外部来宾声纹：为 `null`
- `personType`：
  - `internal`：来自组织成员
  - `external`：自定义外部人员
- `role`：继续承载会议页显示用的职位文案

## 6.3 会议说话人

后续会议里建议逐步收敛到以下口径：

```js
{
  memberId: "mem_xxx" || null,
  voiceprintId: "vp_xxx" || null,
  name: "田中 健太",
  role: "CTO",
  avatar: "田",
  color: "..."
}
```

这样做的好处是：

- 会议里展示仍然保持现有 UI 不变
- 但底层已经拿到稳定主键
- 未来做“按成员搜索参会人”“改名后全局同步”都会更稳

## 7. 管理后台设计口径

## 7.1 页面目标

管理后台成员管理页要从“静态展示”变成“真实通讯录”。

但必须遵守一条原则：

- 页面视觉、结构、文案、当前高保真内容尽量不改
- 只是把背后的成员列表和操作做成真实可用

## 7.2 初始数据要求

当前高保真页面中已有三十多位成员，这批人后续要落成真实 seed 数据。

要求如下：

- 名字、邮箱、角色、部门、状态、最近活跃、加入时间都保持与当前页面一致
- 职位字段全部初始化为空字符串 `""`
- 后续管理员可逐个编辑补齐职位
- 统计卡、筛选 chip、表格行数都继续与当前高保真界面保持一致

## 7.3 成员新增

管理员在“邀请成员”或“新增成员”弹窗中，应可录入：

- 邮箱
- 角色
- 部门
- 职位

其中：

- 职位不在成员表格主列显示
- 但弹窗必须可输入并落库

新增后的成员：

- 会进入真实成员列表
- 可被声纹页点选
- 后续可被会议参会人选择

## 7.4 成员编辑

成员行右侧三点菜单应支持“编辑”和“删除”。

编辑仅允许修改：

- 姓名
- 邮件
- 角色
- 部门
- 职位

注意：

- 表格里不新增“职位”这一列
- 职位只在新增/编辑弹窗中维护

## 7.5 成员删除

成员删除应为真实删除，但要先明确业务口径：

- 如果该成员已绑定内部成员声纹，则成员删除时应同步删除这些声纹记录
- 当前项目口径以“成员主档是内部身份的唯一来源”为准，不再保留解绑转 external 的中间态

也就是说：

- 删成员时，同步删除其绑定的内部成员声纹
- 外部来宾声纹不受影响
- 历史会议如需保留旧文案，后续应依赖已归档快照，而不是依赖仍然存在的声纹记录

这样更符合当前实际产品口径，也能避免成员已删但声纹页仍残留“内部人员壳数据”的混乱。

## 7.6 筛选与搜索

当前管理后台已有以下筛选口径，后续必须做活：

- 全部
- 所有者
- 管理员
- 普通成员
- 待激活
- 已停用
- 部门下拉
- 搜索姓名或邮箱

规则如下：

- `chip` 筛选、部门筛选、搜索词三者可叠加
- 搜索匹配：
  - 姓名模糊匹配
  - 邮箱模糊匹配
- 统计 chip 数字应跟随真实数据刷新
- 表格内容、页脚计数要跟随真实筛选结果变化

## 8. 声纹页设计口径

## 8.1 新增声纹的两种模式

新增声纹时，必须支持两种来源：

### 模式 A：绑定企业内部成员

流程：

1. 用户选择“企业成员”
2. 在成员下拉中搜索并点选一个真实成员
3. 系统自动带出：
   - 姓名
   - 职位
4. 用户录音并生成声纹
5. 保存后，声纹记录绑定该成员 `memberId`

这套模式用于：

- 企业内部正式成员
- 后续会议中需要统一身份识别的人

### 模式 B：自定义外部人员

流程：

1. 用户选择“外部来宾”
2. 手动输入：
   - 姓名
   - 职位
3. 录音并生成声纹
4. 保存后，`memberId = null`，`personType = external`

这套模式用于：

- 客户
- 合作方
- 顾问
- 临时与会嘉宾

## 8.2 声纹编辑

声纹记录必须支持编辑。

可编辑字段：

- 姓名
- 职位

联动规则建议如下：

- 若是 `internal` 声纹：
  - 默认应以成员主档为准
  - 为避免一改两边全乱，MVP 建议：
    - 在声纹页点击编辑 internal 声纹时，实际更新对应成员主档的 `name/title`
    - 然后同步刷新该声纹显示
- 若是 `external` 声纹：
  - 直接修改声纹记录自己的 `name/role`

这样可以避免同一个企业员工在成员页和声纹页改出两个不同名字

## 8.3 声纹下载

“声纹应当可以下载”的实现口径建议明确为：

- MVP 下载的是“声纹记录导出文件”，不是裸 embedding 文本直接展示在界面上
- 导出内容可为 JSON

建议字段：

```json
{
  "id": "vp_xxx",
  "memberId": "mem_xxx",
  "personType": "internal",
  "name": "田中 健太",
  "role": "CTO",
  "sampleCount": 123,
  "createdAt": "...",
  "updatedAt": "...",
  "embedding": [ ... ]
}
```

下载策略：

- 单条声纹支持单独下载
- “导出我的声纹数据”支持批量下载当前账号下全部声纹

## 8.4 声纹搜索

声纹搜索建议支持：

- 姓名模糊匹配
- 职位模糊匹配

后续若存在成员绑定，还可扩展：

- 按 `internal/external` 过滤

但本轮不是必须项

## 9. 成员与声纹的联动规则

## 9.1 内部成员绑定规则

- 一个成员可以有 0 条或多条声纹记录
- 但 MVP 更建议先限制为“一人一条主声纹”
- 如果后续要支持多样本或多版本，再扩展为：
  - 一个成员多条声纹
  - 其中一条为主声纹

本轮建议：

- 同一 `memberId` 默认只保留一条有效声纹
- 新录入时如已存在，则提示“覆盖 / 替换”

## 9.2 成员修改后的同步规则

- 成员页修改 `name/title`
- 若该成员已绑定声纹，则声纹页展示要同步更新
- 会议识别结果里未来新生成的说话人文案，也应读取最新主档

## 9.3 成员删除后的联动规则

- 成员删除时，如存在绑定声纹：
  - 绑定的内部成员声纹同步删除
  - 外部来宾声纹不受影响

## 9.4 外部人员转内部成员

后续可以支持“把外部来宾声纹挂到某个新成员名下”，但本轮不是必须。

本轮仅要求：

- 外部人员声纹可独立存在
- 不强制要求一定绑定组织成员

## 10. 存储方案建议

## 10.1 成员数据

MVP 阶段建议先用本地存储，和当前整体产品口径一致。

建议：

- `localStorage`
- key 示例：`ac_org_members_v1`

原因：

- 当前项目会议、术语、配置大多已走本地存储
- 管理后台成员量级不大
- 更快形成闭环

## 10.2 声纹数据

继续沿用现有 `IndexedDB` 存储声纹 embedding。

原因：

- embedding 数据体量比普通成员资料大
- 现有声纹链路已在 IndexedDB 中运行
- 只需要在现有结构上补字段，不需要推翻重做

## 10.3 种子数据

建议新增“种子初始化”逻辑：

- 首次进入 admin 页时
- 如果当前组织还没有成员数据
- 就按当前高保真界面的三十多位成员初始化真实成员列表

要求：

- 只初始化一次
- 不重复覆盖用户后续手动修改的数据

## 11. 页面与交互修改范围

## 11.1 管理后台

涉及文件建议为：

- `app/admin.html`
- `src/admin/admin-page-app.js`
- `src/storage/org-member-store.js`（新增）

改动方向：

- 保持高保真 HTML 结构与视觉不变
- 把成员表改为真实数据渲染
- 让筛选和搜索真实生效
- 为右侧操作菜单接入：
  - 编辑
  - 删除
- 为新增/邀请弹窗接入真实保存

## 11.2 声纹页

涉及文件建议为：

- `app/voiceprint.html`
- `src/voiceprint/voiceprint-page-app.js`
- `src/storage/voiceprint-store.js`
- `src/storage/org-member-store.js`（复用）

改动方向：

- 新增“内部成员 / 外部来宾”选择
- 新增成员下拉点选
- 自动带出姓名与职位
- 打通编辑、搜索、下载

## 11.3 后续会议页预留

本轮不改会议人员选择，但文档必须明确预留目标：

- 未来会议“选择参会人员”时
- 应直接复用 `org-member-store`
- 不再继续依赖 demo participants 作为长期主数据

## 12. 推荐开发顺序

### 第一步

- 新增成员 store
- 建立成员数据模型
- 做当前高保真成员数据的一次性 seed 初始化

### 第二步

- 把 admin 成员列表改成真实数据渲染
- 做筛选、搜索、统计 chip 真实刷新

### 第三步

- 接通成员新增、编辑、删除
- 弹窗中加入职位字段

### 第四步

- 扩展声纹 store 字段
- 为声纹记录增加 `memberId / personType / companyKey`

### 第五步

- 声纹新增支持“内部成员 / 外部来宾”两种模式
- 内部成员点选后自动带出姓名与职位

### 第六步

- 接通声纹编辑与下载
- 完成成员删除后的声纹联动删除逻辑

## 13. 风险点

- 如果仍然只用姓名做关联，后续改名、重名、会议回放都会出现混乱。
- 如果 internal 声纹允许单独改名而不更新成员主档，会造成成员页和声纹页显示不一致。
- 如果种子数据初始化每次都覆盖，会把后续真实编辑结果冲掉。
- 如果管理后台视觉结构被大改，会偏离当前高保真验收口径。

## 14. 验收口径

## 14.1 管理后台

- 当前三十多位成员已成为真实数据，而不是静态写死 DOM
- 页面视觉与现有高保真界面保持一致
- 搜索姓名或邮箱可真实过滤结果
- 角色筛选、状态筛选、部门筛选可真实生效
- 三点菜单可编辑、删除
- 编辑时可修改：
  - 姓名
  - 邮件
  - 角色
  - 部门
  - 职位

## 14.2 声纹页

- 新增声纹支持“企业成员 / 外部来宾”两种模式
- 选择企业成员后，姓名和职位自动填入
- 外部来宾可自定义姓名和职位
- 声纹记录支持编辑姓名和职位
- 声纹记录支持单独下载
- 批量导出声纹数据可用

## 14.3 联动关系

- 成员修改姓名/职位后，已绑定声纹展示同步更新
- 删除成员后，已绑定的内部成员声纹会同步删除
- 后续会议页可直接复用成员主档，不需要再另造一套人名数据

## 15. 最终结论

这部分后续开发的核心不是“把两个页面分别做活”，而是：

- 先建立统一的“人员主档”
- 再让声纹成为“挂在人员主档上的识别资产”
- 同时保留“外部来宾”这条不绑定内部成员的灵活路径

如果按这个口径做，后面你再要求：

- 会议中搜索真实参会人员
- 会议里自动回填真实身份
- 成员改名后全局同步

都会顺很多，不需要推翻重做。

## 16. 会议页真实参会人员搜索开发修改点清单

### 16.1 目标

本节只用于后续“会议创建弹窗 / 会议页参会人选择”开发拆单，不涉及本次代码修改。

目标效果如下：

- 参会人输入框不再是简单下拉，而是可搜索的候选框。
- 搜索支持泛匹配，至少覆盖：
  - 姓名
  - 职位
  - 部门
  - 邮箱
- 用户在输入框中先选中一个候选人，该人只会暂时停留在输入框中。
- 只有点击右侧“添加参会人”按钮后，才会真正进入下方已选参会人列表。
- 已选参会人颜色复用组织成员主档中的 `avatarColor`。
- 会议界面发言气泡颜色与参会人颜色保持一致。
- “添加参会人”按钮改为当前界面中的深蓝主按钮样式，以提高可见性。

### 16.2 统一产品口径

- 会议参会人选择后续必须直接复用 `org-member-store`，不再继续依赖 demo participants 作为长期主数据。
- 会议创建弹窗内的候选项来源限定为当前组织下的真实成员。
- 第一版建议仅允许添加 `status = active` 的成员进入会议参会人列表。
- 候选搜索框与最终已选列表是两层状态，不可混在一起：
  - 搜索框只承载“当前候选”
  - 下方 tag 列表才承载“已正式加入会议的参会人”
- 已经添加进下方列表的人，不应重复添加。

### 16.3 统一数据结构

后续建议会议创建时写入的 participant 对象统一为：

```js
{
  memberId: "mem_xxx",
  source: "org_member",
  companyKey: "神泉科技",
  name: "李明",
  title: "销售总监",
  department: "销售",
  email: "liming@shenchuan.com",
  avatar: "李",
  color: "var(--speaker-1)"
}
```

字段口径：

- `memberId`：会议参会人与成员主档的稳定关联主键。
- `source`：固定写为 `org_member`，用于和旧 demo 参会人结构区分。
- `companyKey`：当前组织作用域。
- `title / department / email`：供搜索、展示、后续总结或导出复用。
- `avatar`：复用成员主档 `avatarText`。
- `color`：复用成员主档 `avatarColor`。

#### A. `app/home.html`

#### B. `src/home/home-page-app.js`

改什么：

- 把当前 `DEMO_MEETING_PARTICIPANTS` 作为候选来源的逻辑，改为 `getOrgMembersByCompany(currentUser.company)`。
- 新增会议创建弹窗内的成员搜索、候选过滤、候选点击、正式添加、移除已选等状态控制。

不改什么：

- 不改会议标题、语言配对、创建确认逻辑。
- 不在这一层直接改会议页渲染。

- 候选列表由真实成员主档计算得到
- 已选列表项使用标准 participant 对象

风险点是什么：

- 如果这里只改搜索 UI、不改 payload 输出结构，会议页仍会退回到 demo 逻辑。
- 如果默认“当前用户”仍沿用 demo 名称匹配，后续会和真实成员主档脱节。

#### C. `src/storage/org-member-store.js`

改什么：

- 补一个前端 helper 也可接受，例如：
  - `searchOrgMembers(companyKey, keyword, options)`
  - 或在 Home 层自行筛选，但底层字段必须完整可读。

不改什么：

- 不改现有成员 CRUD 基本结构。
- 不改成员主档主键生成逻辑。

数据结构怎么定：

- 成员主档继续作为颜色与身份的唯一来源：
  - `avatarText`
  - `avatarColor`
  - `title`
  - `department`

风险点是什么：

- 如果新增成员不自动分配颜色，后续会议参会人和发言气泡会退化成灰色兜底，无法满足“人物颜色全链路统一”。

#### D. `src/storage/meeting-store.js`

改什么：

- 调整 `normalizeParticipant/normalizeParticipants`，优先识别真实组织成员 participant 结构。
- 若存在 `memberId`，优先保留其 `name/title/avatar/color`，不要再优先回退到 demo participant。
- 创建会议、保存 segments、结束归档时，participants 合并逻辑要继续保留 `memberId`。

不改什么：

- 不改会议归档与恢复入口。

数据结构怎么定：

风险点是什么：

- 如果 meeting-store 仍强依赖 `getDemoParticipantByName`，会议恢复、归档、回放时会丢失真实成员颜色与主键关联。

#### E. `src/meeting/meeting-page-app.js`

改什么：

- 调整 `normalizeMeetingParticipants()` 与 `getSpeakerPool()`，优先读取 meeting record 中已经写好的真实 participant 对象。
- 会议页 legend、发言气泡、统计卡片继续沿用 `speaker.color`，但这个颜色要来自真实 participant.color。

不改什么：

- 不改当前会议页整体布局。
- 不改 transcript / summary / archive 主流程。
- 不在第一版里做复杂“同名人合并”或“多身份映射”。

数据结构怎么定：

  - `memberId`
  - `name`
  - `role`
  - `avatar`
  - `color`

不改什么：

### 16.6 验收口径

- 会议创建弹窗中的参会人输入框支持按姓名、职位、部门、邮箱实时搜索。
- 用户点击候选人后，该人先固定在输入框中，不会自动进入下方列表。
- 只有点击“添加参会人”按钮后，候选人才能进入下方已选列表。
- 已添加成员不会重复添加。
- 已选参会人 tag 颜色与组织管理中的该成员头像颜色一致。

- 新增成员后，若被加入会议参会人，其颜色不再退化为灰色兜底。
- 会议创建、会议恢复、会议归档后，participant 的 `memberId` 仍能保留，不会丢失真实身份关联。
