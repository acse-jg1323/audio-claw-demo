# 系统界面多语言与 AI 语言联动最终开发清单

> 版本：v1.0
> 日期：2026-06-29
> 用途：作为当前分支的最终开发与验收口径，直接指导代码落地与 demo 走查。

## 一、最终口径

本轮已经进入真实开发阶段，不再只是需求整理。

本轮 demo 目标有两条主线：

- 系统界面支持三种语言：简体中文、日文、English
- 会议前、中、后相关 AI 能力的输出语言，跟随当前系统语言

这里的“系统语言”指用户在设置页中主动选择的界面语言。后续文档中统一称为：

- `systemLanguage`
- 可选值：`zh` / `ja` / `en`

本轮最终确认口径如下：

- `systemLanguage` 是系统界面语言、主阅读语言默认值、AI 助手输出语言、AI 总结输出语言的统一源头
- `mainLanguage` 跟随 `systemLanguage`
- `meetingTargetLanguage` 暂时不自动改，仍然保留为用户手动选择
- 所有客户可见说明性文案都要翻译
- 术语条目值、术语源词、译法、人名、公司名等业务资产值不翻译
- 必须保证中 / 日 / 英切换后页面样式不被撑坏

## 三、核心需求结论

### 1. 系统设置页要真实接入界面语言

在 `app/settings.html` 中，现有“界面语言”区域不再只是原型展示，而要真实接入：

- 简体中文
- 日文
- English

并要求支持热切换：

- 用户在设置页切换后，当前页面文案立即切换
- 跳转到其他页面后继续保持已选择语言
- 刷新页面后继续保持已选择语言

本轮不需要支持：

- 繁体中文
- 韩文
- 更多外语

即本轮只保留 demo 必需的三种语言。

### 2. 系统语言切换后，主阅读语言要联动

会议页当前已有“主阅读语言”配置。根据本轮需求，切换系统语言后：

- 默认主阅读语言要变为当前 `systemLanguage`
- 会议前新建会议时，默认带入当前 `systemLanguage`
- 会议进行中如果用户切换系统语言，当前会议页的主阅读语言也要同步切换
- 会议结束后的总结查看态、历史回看态，也要按当前 `systemLanguage` 显示对应可展示内容

这条需求的业务含义是：

- 当前用户把系统切到哪种语言，系统就默认把那种语言视为这个用户的母语阅读语言
- 因此识别后的主显示语言、回译目标、总结阅读语言，都应围绕这个语言工作

### 3. AI 助手输出语言要跟随系统语言

AI 助手模块需要满足：

- 不管用户用什么语言提问
- AI 助手最终输出都必须跟随当前 `systemLanguage`
- 不额外增加第二次模型调用
- 通过 prompt 约束完成

实现口径按你的要求整理为：

- AI 助手的系统 prompt 需要准备三套语言版本：中文 / 日文 / 英文
- 当 `systemLanguage !== zh` 时，不是让模型先翻译后再回答，而是直接使用对应语言版本的 prompt
- 同时在 prompt 中明确补一句强约束，语义等价于：
  - 由于当前客户使用的语言是 `xxx`，你应该使用 `xxx` 语言输出全部答案

需要强调：

- 为了保证解析稳定性，AI 助手返回的 JSON 结构字段名建议继续保持固定，不要跟着系统语言变化
- 变化的是 prompt 说明文字和最终 `answer` 内容语言
- `citations` 里的片段 id 也保持原结构，不做语言化

### 4. AI 会议总结输出语言要跟随系统语言

AI 会议总结模块需要满足：

- `summaryLanguage` 跟随当前 `systemLanguage`
- 会议总结的系统 prompt、生成约束、风格要求，都要有三套语言版本
- 不管会议原始发言是什么语言，最终总结都必须输出为当前系统语言
- 会议总结标题、状态描述、导出文案，也要跟着系统语言走

同样建议：

- Summary Agent 的 JSON schema 字段名继续保持稳定
- 只切换 prompt 说明语言和模型最终产出的正文语言
- 这样可以避免前端解析层因为字段翻译而失效

## 四、翻译范围

### 1. 需要翻译的页面范围

本轮涉及的客户可见页面，原则上都需要翻译其可见文案：

- `app/login.html`
- `app/home.html`
- `app/index.html`
- `app/settings.html`
- `app/admin.html`
- `app/voiceprint.html`
- `app/terms.html`

其中：

- `terms.html` 要翻译所有按钮、说明、筛选、空态、提示、弹窗等说明性文字
- 但术语条目本身的数据内容不要求翻译

### 2. 术语热词库页面的特殊边界

术语热词库页面按以下边界处理：

- 术语名称、源词、别名、译法、语言绑定的数据内容，不做自动翻译
- 因为这些内容本身就是业务资产，且与语言对绑定
- 但是页面中所有客户可见的说明性文字必须翻译

包括但不限于：

- 页头标题、副标题
- Tab 名称
- 搜索框 placeholder
- 筛选项名称
- 按钮文案
- 表头文案
- 空状态文案
- 导入导出提示
- 权限提示
- 弹窗说明文字
- 错误提示和成功提示

### 3. 需要翻译的运行时文案

不能只翻静态 HTML，还要覆盖运行时 JS 动态写入文案，包括但不限于：

- 按钮状态文案
- loading 文案
- 空态文案
- 错误提示
- `alert` / `confirm`
- placeholder
- tooltip / title
- status badge
- 回看模式说明
- AI 助手欢迎语
- AI 助手快捷指令 chips
- AI 总结状态栏文案
- 导出标题和导出文本中的固定中文

## 五、AI 语言联动规则

### 1. 统一原则

本轮建议将 `systemLanguage` 作为单一事实来源，至少驱动以下三个维度：

- 界面显示语言
- 主阅读语言
- AI 输出语言

### 2. 会议页中的联动关系

会议页当前已有：

- `mainLanguage`
- `meetingTargetLanguage`

本轮先明确以下强约束：

- `mainLanguage` 默认跟随 `systemLanguage`
- AI 助手输出语言跟随 `systemLanguage`
- AI 会议总结输出语言跟随 `systemLanguage`

这里的最终口径已经确认：

- 只让 `mainLanguage` 跟随系统语言
- `meetingTargetLanguage` 继续手动配置
- 系统语言切换时，不自动覆盖用户对 `meetingTargetLanguage` 的选择

### 3. AI 助手语言规则

AI 助手按以下规则执行：

- 输入问题可以是中文、日文、英文中的任意一种
- 助手回答必须使用当前 `systemLanguage`
- 助手引用会议片段时，继续引用真实片段 id，不改结构
- 如果当前转写不足，拒答文案也必须使用当前 `systemLanguage`

推荐实现方式：

- 保留现有接口结构
- 将 `ASSISTANT_SYSTEM_PROMPT` 改成按语言返回
- 保留 `answer_language` 字段，但其值来自 `systemLanguage`
- 额外补充一句输出语言强约束

### 4. AI 总结语言规则

AI 会议总结按以下规则执行：

- `summaryLanguage` 固定取当前 `systemLanguage`
- 总结正文、决议、待办、遗留问题、置信说明，全部输出当前语言
- “会议总结”标题后缀、状态栏、导出内容中的固定文字也必须本地化
- 负责人姓名不翻译，按会议中展示名保留

推荐实现方式：

- 将 `SUMMARY_SYSTEM_PROMPT` 改成按语言返回
- `summary_language` 继续作为结构化字段传给模型
- 模型返回 JSON 的 key 不要翻译

## 六、样式与交互约束

这是本轮 demo 的硬性要求：

- 翻译后所有页面样式必须基本保持不变
- 不能因为英文或日文文案变长，把布局撑坏
- 不能出现按钮被撑开后换行错位
- 不能出现卡片高度明显异常
- 不能出现表头、标签、tab、侧边栏名称错位

开发时必须重点调试以下场景：

- 左侧主导航
- 设置页二级导航
- 会议页顶部配置区
- 会议总结工具栏按钮
- AI 助手抽屉标题、快捷指令、输入框 placeholder
- 术语页与声纹页表头
- 管理页表格、按钮、权限标签
- 空状态和错误状态

推荐的样式守则：

- 尽量保持现有布局结构，不为多语言单独重做版式
- 对容易变长的文案预留最小宽度和最大宽度
- 必要时对按钮、标签、标题采用更短的英文/日文文案
- 对超长文本增加截断、省略号或换行策略
- 避免把可变长文案直接写死在固定宽度容器里
- 中/日/英三种语言都要做实际页面走查

## 八、建议的数据与状态设计

### 1. 建议新增系统语言持久化

当前仓库里还没有真实的 `uiLocale` 或 `systemLanguage` 存储。

建议新增单独存储，例如：

- `localStorage` key：`ac_system_language`

默认值：

- `zh`

### 2. 建议保存口径

建议至少满足以下行为：

- 未登录时也能读取到最近一次选择的语言
- 登录后各页面都读取同一个值
- 切换后立刻写入持久化

如果后续要更精细，也可以再区分：

- 设备级语言偏好
- 用户级语言偏好

但本轮 demo 没必要上升到复杂账户设置模型。

### 3. 建议事件联动

为了支持热切换，建议页面内至少有一种联动机制：

- 切换语言后，当前页面立即重渲染
- 其他页面在初始化时读取最新语言

如果需要更顺畅，也可以加：

- 自定义 `window` 事件
- 或轻量的订阅发布机制

## 十一、开发守则

### 1. 不要把多语言实现成到处写 `if`

必须尽量抽到统一词典和统一工具层，避免：

- 页面里散落大量 `if (lang === "ja")`
- 后续维护困难
- 漏翻概率高

### 2. 结构化字段保持稳定

AI 助手与 AI 总结的结构化输出：

- JSON key 不翻译
- 字段结构不变化
- 只改 prompt 文案和最终内容语言

这样最稳。

### 3. 不翻译业务资产值

本轮不应翻译：

- 术语数据内容
- 人名
- 公司名
- 会议原始转写片段
- 引用 id

本轮翻的是“系统界面”和“AI 输出语言”，不是改写业务源数据。

### 4. 保证热切换是真热切换

最低要求：

- 当前页切换后立即生效
- 不要求用户手动刷新才看到变化

### 5. 样式优先稳，不为 demo 做大改版

优先原则：

- 保持当前页面布局
- 通过文案长度控制、宽度约束、截断策略解决问题
- 不轻易重做整页结构

### 6. 默认值口径要统一

本轮建议统一默认：

- `systemLanguage` 默认 `zh`
- `mainLanguage` 默认跟随 `systemLanguage`
- `meetingTargetLanguage` 默认保持现有手动值，不做联动覆盖
- AI 助手输出语言跟随 `systemLanguage`
- AI 总结输出语言跟随 `systemLanguage`

## 十二、验收标准

### 1. 设置页

- 可真实切换简体中文 / 日文 / English
- 切换后当前页立即生效
- 刷新后语言保持

### 2. 全站界面

- 登录页、主页、会议页、设置页、管理页、声纹页、术语页说明文案都可切换
- 无明显漏翻
- 无明显中英日混杂

### 3. 会议页联动

- 切换系统语言后，主阅读语言同步变化
- 切换系统语言后，`meetingTargetLanguage` 保持用户手动值不变
- AI 助手回答语言同步变化
- AI 会议总结输出语言同步变化

### 4. 样式稳定

- 中/日/英三套语言下，页面布局无明显错位
- 按钮、表头、标签、导航不被撑坏

### 5. Terms 页边界正确

- 术语内容本身不被翻译
- 说明性 UI 文案全部可翻译

## 十三、当前建议结论

本轮 demo 最稳的落地方式是：

- 先把 `systemLanguage` 作为统一源头引入
- 先做设置页真实切换
- 再把会议页的 `mainLanguage`、AI 助手、AI 总结都挂到这个统一语言状态上
- `meetingTargetLanguage` 保持手动，不参与自动联动
- 最后补齐各个页面和运行时动态文案

其中最关键的两条开发纪律是：

- 不要只翻静态 HTML，一定要连 JS 运行时文案一起翻
- 一定要做中/日/英三语样式走查，保证 demo 页面观感不变
