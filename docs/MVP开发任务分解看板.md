### 当前正式链路与选型结论

阶段三当前已经明确采用“识别后路由翻译”正式链路：

- `DeepThink WebSocket`：只负责实时原文转写 ⚠️：现在Deepthinking模型不再承担任何target\_language 翻译任务！翻译文本仅由senseaudio-s2-lite低延迟生成（均0.3s）
- `规则语言判别模块`：产出 `source_language`
- `翻译路由层`：产出 `target_language`
- `senseaudio-s2-lite`：做逐 segment 文本翻译
- `页面层`：先显示原文，再异步补译文

当前模型选型结论：

- `senseaudio-s2-lite`：作为 MVP 会中逐 segment 翻译的默认模型✅
- `senseaudio-s2`：作为质量上限参考或后续会后重翻备选
- `senseaudio-s2-flash`：当前不建议作为默认链路，原因是两轮实测均出现明显长尾延迟

### MVP 语言判别阈值方案

为保证后续“识别后路由翻译”稳定可控，MVP 阶段先采用轻量规则判别原始语言，不引入额外语言识别模型。

当前判别目标语言仅聚焦：

- `zh`
- `ja`
- `en`
- `unknown`

#### 一、判别优先级

建议按以下优先级顺序判断：

1. 先判定是否包含明显日语字符✅
2. 再判定是否属于高纯度英文句子✅
3. 再判定是否属于中文主导句子✅
4. 最后进入 `unknown` 兜底✅纯数字、杂字符等，不翻译

#### 二、日语判别规则

如果一句话中出现以下任一情况，优先判定为 `ja`：

- 出现平假名
- 出现片假名
- 出现典型日语助词或词尾表达，例如：`です`、`ます`、`でした`、`してください`

说明：

- 中日的汉字有重叠，因此不能只看 CJK 字符
- 只要出现明显日语假名，基本可以直接判定该句原始语言为日语
- 日语判别优先级高于中文

#### 三、英文判别规则

英文由于可能以术语、品牌名、缩写形式混入中文或日文句子，因此英文判别阈值应明显更高。

MVP 建议规则如下：

- 仅当句子中 `英文字符占比 >= 95%` 时，判定原始语言为 `en`
- 若一句话仅包含少量英文单词、产品名、缩写、品牌名，不判定为英文原始语言
- 当主阅读语言不是英文时，只有高纯度英文句子才进入“翻译回主阅读语言”的逻辑

说明：

- 这样可以避免把“OK 这个 API 先走 websocket”“AudioClaw 这个 demo 很稳定”这种中英混说句子误判成英文
- 英文在商务会议中常作为夹杂词出现，因此必须采用高阈值保守判定

#### 四、中文判别规则

若一句话满足以下条件，则判定为 `zh`：

- 未触发日语判别
- 未触发英文高阈值判别
- 句子主体为中文字符

说明：

- 中文作为当前 MVP 默认主阅读语言，可采用“宽判定”策略
- 也就是说，只要不是明显日语、也不是高纯度英文，就优先归入中文

#### 五、unknown 兜底规则

以下情况进入 `unknown`：

- 字符过短，无法判别
- 主要由数字、符号、缩写组成
- 同一句中多语种混杂且没有明显主导语言
- 单个英文字母、数字或其他噪音型单字符，不做捞回，仍保持 `unknown`

MVP 阶段建议：

- `unknown` 默认按“非主阅读语言”处理，翻译为主阅读语言
- 前端保留日志字段，便于后续调试误判样本

#### 六、MVP 阶段的路由逻辑

在完成语言判别后，翻译路由按以下规则执行：

- 若 `source_language = 主阅读语言`
  - 翻译为 `会议外语目标语言`
- 若 `source_language != 主阅读语言`
  - 翻译为 `主阅读语言`

以当前默认会议配置为例：

- 主阅读语言：`zh`
- 会议外语目标语言：`en` 或 `ja`

则结果为：

- 中文句子 -> 翻译为本场会议目标外语
- 日语句子 -> 翻译为中文
- 高纯度英文句子 -> 翻译为中文
- 无法明确判别的句子 -> 暂按翻译为中文处理

#### 七、当前实现原则

- 先保证规则简单、透明、可解释
- 先尽量避免把中文/日文误判为英文
- 优先支持中日会议和中英会议
- 后续若误判样本积累较多，再升级为“规则 + 模型”的混合判别方案

- 用户可以在会议中看到稳定的双语结果✅
