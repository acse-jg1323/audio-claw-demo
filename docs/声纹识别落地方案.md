# 声纹识别落地方案（会议中自动标注说话人身份）

> 版本：v0.1
> 目标：把当前会议页里"假的说话人占位"替换为真实声纹识别，实现会议中自动标注"这句话是谁说的"。闭集、浏览器端、无后端。

## 一、当前结论

- 当前会议页的"说话人"是**假数据**：`meeting-page-app.js` 的 `pickDemoSpeaker()` 按 segment 序号轮流分配演示参会者，与声音无关（气泡里甚至标着"演示占位"）。
- 声纹库页 `voiceprint.html` 目前是**纯静态外壳**：6 个人写死在 HTML，"添加声纹"弹窗按钮无逻辑，上传区是装饰。
- 但底层数据链路**可复用**：浏览器已采 16kHz PCM（`audio-pcm-capture.js` 的 `onChunk`），ASR 已回传 `segmentId` + `timestampEnd`（`senseaudio-websocket-client.js`）。
- SenseAudio 平台**无声纹产品**（全量文档索引已确认），声纹能力必须自建。
- 采用**浏览器端方案**：用 `onnxruntime-web` 在前端跑 CAM++ 声纹模型，声纹不出设备，契合 MVP 架构与 `voiceprint.html` 的"端到端、不出设备"承诺。

## 二、方案选型

| | 方案一：浏览器端（采用） | 方案二：Python 后端（不采用） |
|---|---|---|
| 模型 | CAM++ ONNX + onnxruntime-web (WASM) | 3D-Speaker (Python) |
| 跑在哪 | 用户浏览器 | 自建服务器 |
| 改动 | 纯前端，不破坏 Vercel 部署 | 新增后端工程 |
| 声纹库 | IndexedDB | 后端存储 |
| 隐私 | 声纹不出设备 | 上传服务器（与页面承诺冲突） |
| 契合度 | 契合当前 MVP | 偏离 MVP |

**结论：方案一。** 无 GPU 不是问题——声纹只在每句话结束时跑一次，不是连续推理；CPU/WASM 提取一段 embedding 约 100–300ms，跟得上会议节奏。

## 三、模型

- 文件：`app/models/campplus_cn_common.onnx`（约 **30MB**，权重内嵌单文件；由 CAM++ 官方 PyTorch 权重导出并校验，ONNX vs PyTorch 误差 9.7e-06。注意：导出默认会把权重外置成 `.onnx.data`，浏览器端 onnxruntime-web 无法加载外置权重，必须内嵌成单文件）。
- 输入：`feats`，形状 `[batch, 帧数, 80]`，即 80 维 FBank 特征。
- 输出：`embedding`，形状 `[batch, 192]`，即 192 维声纹向量。
- 比对：**余弦相似度**。判定阈值 **0.31**（CAM++ 官方 `yesOrno_thr`）。≥ 阈值且为最高分 → 命中该身份；否则标记"未识别"。

## 四、FBank 特征参数（最关键，JS 必须严格对齐）

CAM++ 用 Kaldi 默认 FBank 参数（`torchaudio.compliance.kaldi.fbank`），仅显式指定 80 mel。JS 端实现必须逐项对齐，否则识别全错：

- 采样率：16000 Hz
- 窗长：25ms（400 点）
- 帧移：10ms（160 点）
- mel 滤波器数：80
- 加窗：povey 窗（Kaldi 默认，非 Hann）
- 预加重：0.97
- dither：推理时设 0（关闭，保证可复现）
- 去直流：每帧减去均值（remove_dc_offset，Kaldi 默认开）
- 能量项：use_energy=False
- 输入波形幅度：`[-1, 1]` 浮点
- 实例归一化：`mean_nor=True`——整段特征沿时间轴减均值（CMN）

**验证方法**：用 `models/_export/` 下的 PyTorch 环境对同一段 wav 跑官方 FBank+模型得到 embedding，与 JS 端结果对拍，余弦相似度需 > 0.99 才算特征实现正确。这是本方案唯一高风险点，必须先过这一关再继续。

## 五、改动文件清单（增量，不破坏现有样式与链路）

**新增**
- `app/models/campplus_cn_common.onnx` —— 模型文件（已就绪）。
- `src/integrations/voiceprint-engine.js` —— 核心引擎：加载 onnx、PCM→FBank、提 embedding、余弦比对。
- `src/storage/voiceprint-store.js` —— IndexedDB 增删查声纹，对标现有 `meeting-store.js`。

**修改**
- `voiceprint.html` —— "添加声纹"弹窗接真录音+注册；人员列表改为从 IndexedDB 渲染。
- `src/meeting/meeting-page-app.js` —— `onChunk` 里加 PCM 环形缓冲；把 `pickDemoSpeaker()` 换成声纹比对（异步补标）。

**依赖**
- 引入 `onnxruntime-web`（CDN 或本地）。FBank 用纯 JS 实现，不引重型音频库。

## 六、实时链路与"延迟补标"交互

```
PCM onChunk ──┬─► 发给 ASR WebSocket ──► result_final {text, segmentId, timestampEnd}
              │                                              │
              └─► 写入 PCM 环形缓冲 ◄─────── 按 timestamp 切出该句音频
                                                             │
                                          FBank → embedding → 余弦比对声纹库
                                                             │
                                                身份 / "未识别" ──► 更新气泡
```

ASR 回 `result_final` 时该句已说完，故身份**延迟半句到一句补标**：气泡先显示"识别中…"，拿到身份后替换为姓名+头像。此体验为行业通用（飞书妙记、腾讯会议同理），可接受。

## 七、声纹库数据结构（IndexedDB）与隐私

```
{
  id: string,          // uuid
  name: string,        // 真实姓名
  role: string,        // 角色/部门（可选）
  color: string,       // 气泡配色，复用现有调色板
  embedding: number[], // 192 维，注册时多段取平均，提升稳定性
  createdAt: number
}
```

- 注册建议录 10–20 秒，切多段分别提 embedding 后取平均（L2 归一化后平均），降低单段噪声影响。
- 声纹向量仅存本地 IndexedDB，不上传，符合页面"端到端、不出设备、不用于训练"承诺。

## 八、分步实施计划与验收

1. **放模型 + 引擎骨架**：onnx 入 `app/models/`，写 `voiceprint-engine.js` 的加载与 onnx 推理。验收：能 fetch 模型并对一段假特征跑出 192 维向量。
2. **FBank 对齐（高风险关卡）**：JS 实现 FBank，与 PyTorch 对拍。验收：同一 wav，JS embedding 与官方 embedding 余弦 > 0.99。
3. **声纹库存储**：`voiceprint-store.js` + IndexedDB 增删查。验收：注册/读取/删除可用。
4. **注册流程**：接 `voiceprint.html` 录音→提 embedding→存库→列表渲染。验收：真实添加一个人并显示。
5. **会议接入**：PCM 环形缓冲 + 替换 `pickDemoSpeaker` + 延迟补标。验收：两人对话，气泡正确标注各自姓名，未登记者显示"未识别"。
6. **联调与阈值校准**：用真实多人音频校准 0.31 阈值，平衡误判/漏判。

## 九、风险与边界

- **第 2 步 FBank 对齐是成败关键**，必须先过对拍再往下做。
- 闭集前提：仅识别已登记人员；未登记者标"未识别"，不做自动建档。
- 重叠说话/抢话场景本方案按"一句话=一个人"近似处理（跟随 ASR 的 VAD 断句），暂不做句内说话人切分；后续如需可引入 pyannote 类实时 diarization，属二期。
