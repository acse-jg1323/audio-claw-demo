# 翻译模块设计草案

> 适用阶段：智能会议系统 MVP / POC
> 当前目标：在不破坏会中实时性的前提下，实现可扩展、可热切换、可解释的双语翻译展示能力
> 当前背景：ASR 已采用 SenseAudio `DeepThink` WebSocket 跑通实时转写主链路

## 一、设计目标

当前会议系统的翻译模块，不应简单理解为“ASR 顺便翻一下”，而应作为会中核心体验模块独立设计。

本模块的主要目标有四个：

- 保证会中页面始终优先稳定展示原文
- 在当前模型能力限制下，尽量实现双语对照和热切换体验
- 为后期中英、中日、多语混说场景预留扩展空间
- 避免把整套产品逻辑强绑定在 `DeepThink WebSocket` 的固定 `target_language` 上

## 二、当前现状与问题

### 1. 当前已跑通的能力

- 已跑通 `DeepThink WebSocket` 实时任务
- 已实现麦克风采集、流式识别、分段结果返回、结束任务
- 已验证服务端可在识别结果中返回翻译字段

### 2. 当前明确的模型边界

当前 `DeepThink WebSocket` 的翻译能力，更适合：

- 固定目标语言
- 会中快速展示
- 同一场任务内持续输出原文和译文

当前不应假设它天然支持：

- 同一条 WebSocket 任务内，不同 segment 动态切换 `target_language`
- 复杂多语混说下的双向自动反向翻译
- 完整可控的翻译策略配置

### 3. 当前产品上的核心矛盾

产品希望实现的效果是：

- 说中文时，显示外语译文
- 说外语时，显示中文译文
- 后续可支持不同会议目标语言
- 最好支持热切换

但当前 `DeepThink WebSocket` 的 `target_language` 更像任务级固定参数，不适合直接承担上述全部职责。

因此，需要在 ASR 之上额外补一层“翻译路由逻辑”。

## 三、总体设计思路

本模块采用四层结构：

- 实时转写层
- 语言判断层
- 翻译路由层
- 结果展示层

这四层共同构成“识别后路由翻译”方案。

## 四、什么是识别后路由翻译

识别后路由翻译，指的是：

1. 先用 ASR 把语音转换为原文文本
2. 再判断这条原文的语言
3. 再根据本场会议的语言规则，决定它应该翻译成什么语言
4. 最后再调用翻译模块，补上译文展示

这意味着：

- ASR 负责“听懂并写出原文”
- 翻译模块负责“根据会议规则决定怎么翻”

这种方案的好处是：

- 不依赖单一模型是否支持复杂双向翻译
- 可以支持后续热切换
- 可以替换翻译执行能力
- 原文和译文逻辑分离，更便于调试和演进

## 五、四层结构说明

## 1. 实时转写层

### 作用

- 接收麦克风音频
- 调用 `DeepThink WebSocket`
- 输出实时 `segment` 原文结果

### 输入

- 麦克风音频流
- ASR 连接配置
- 会中状态控制

### 输出

- `segment_id`
- `text`
- `timestamp_start`
- `timestamp_end`
- `is_final`
- 可选的模型原生翻译字段

### 当前实现状态

- 已完成基本验证

### 当前职责边界

- 只保证“原文转写”稳定
- 不承担最终翻译策略决策

## 2. 语言判断层

### 作用

- 判断每条原文 `segment` 属于哪种语言

### 输入

- 单条 `segment` 原文文本

### 输出

- `source_language`

### 当前建议实现方式

MVP 先采用轻量规则判断模块，例如：

- 全中文字符为主，判定为 `zh`
- 含明显日语假名，判定为 `ja`
- 含英文单词且比例明显较高，判定为 `en`
- 无法判定时，进入兜底策略

### 说明

- 这层后续可以升级为更稳的语言识别器
- 现阶段先采用规则法，便于快速落地和可解释调试

## 3. 翻译路由层

### 作用

- 根据会议设置和原文语言，决定当前这条 `segment` 应翻译成什么语言

### 输入

- `source_language`
- 会议主阅读语言
- 会议外语目标语言
- 当前翻译模式

### 输出

- `target_language`
- 是否需要翻译
- 使用哪一类翻译执行器

### 核心思想

这层不负责真正翻译文本，它只负责做决策。

### MVP 推荐规则

当前建议采用以下规则：

- 如果 `source_language = 主阅读语言`
  - 翻译为 `会议外语目标语言`
- 如果 `source_language != 主阅读语言`
  - 翻译为 `主阅读语言`

### 以当前默认会议为例

- 主阅读语言：`zh`
- 会议外语目标语言：`ja`

则规则为：

- 中文原文 -> 翻译为日语
- 非中文原文 -> 翻译为中文

### 为什么不直接写死“非中文一定翻译成中文”

因为更好的产品抽象应是：

- 主阅读语言
- 会议外语目标语言

这样未来：

- 中文团队主导会议
- 日语团队主导会议
- 英语主界面会议

都可以复用同一套逻辑。

## 4. 结果展示层

### 作用

- 先展示原文
- 再展示译文
- 翻译失败时也不阻塞原文展示

### 输入

- 原文 `segment`
- 译文结果
- 当前会议展示模式

### 输出

- 会中页面气泡内容

### MVP 展示原则

- 原文优先
- 译文补充
- 译文允许稍后到达
- 翻译失败时，仍保留原文

## 六、会议设置设计

为了让翻译模块具备可配置性和热切换能力，建议在会议维度保留以下设置。

### 1. 主阅读语言

定义：

- 当前界面主要服务哪种语言的用户阅读

建议：

- 默认跟随界面语言

MVP 默认值：

- `zh`

### 2. 会议外语目标语言

定义：

- 当前会议中，主阅读语言之外，重点需要互译的另一种语言

MVP 首选项：

- `ja`
- `en`

### 3. 翻译模式

建议预留以下模式：

- `固定目标语言模式`
- `双语路由模式`
- `仅原文模式`

其中当前重点是：

- `双语路由模式`

### 4. 是否允许热切换

建议：

- 会议过程中允许修改“会议外语目标语言”
- 热切换仅影响切换后的新 segment
- 已生成的历史 segment 不强制回刷

这样可以兼顾：

- 交互稳定性
- 实现复杂度
- 用户理解成本

## 七、每条 segment 的决策流程

每条语音分段到达后，建议按以下顺序处理：

1. `DeepThink WebSocket` 返回原文 `segment`
2. 页面立即展示原文气泡
3. 语言判断层识别 `source_language`
4. 翻译路由层判断是否需要翻译
5. 若需要，则生成 `target_language`
6. 将原文提交给翻译执行器
7. 翻译结果返回后，补写到当前气泡的译文区域
8. 若翻译失败，则保留原文，译文区域可为空或显示失败状态

## 八、识别后路由翻译接口层设计

为了让“识别后路由翻译”真正落地，建议在前端与翻译执行器之间定义一个明确的接口层，而不是在页面代码中临时拼 prompt。

### 1. 接口层职责

- 接收 ASR 返回的单条 `segment`
- 接收语言判断层输出的 `source_language`
- 接收翻译路由层输出的 `target_language`
- 组装翻译请求
- 调用具体翻译执行器
- 把翻译结果回填到页面当前气泡

### 2. 输入结构

建议输入结构至少包含：

- `segment_id`
- `text`
- `source_language`
- `target_language`
- `main_language`
- `meeting_target_language`
- `context_tail`
- `glossary_terms`

其中：

- `text`：当前这条 segment 的原文文本
- `context_tail`：前 1-2 条已识别完成的 segment 原文，用于帮助翻译器理解指代、省略和上下文
- `glossary_terms`：后续可选，用于术语约束

### 3. 输出结构

建议输出结构至少包含：

- `segment_id`
- `translated_text`
- `target_language`
- `translator_model`
- `latency_ms`
- `status`

### 4. context_tail 的使用原则

`context_tail` 是当前方案中一个比较重要的增强项，建议保留。

推荐做法：

- 默认传前 `1-2` 条 segment
- 不传太长，避免增加时延和干扰
- 当前条原文仍应作为翻译核心输入
- `context_tail` 只帮助模型理解上下文，不要求模型重新翻译历史内容

适用价值：

- 帮助理解代词
- 帮助理解省略主语或宾语的短句
- 帮助会议口语中的承接关系

### 5. Prompt 设计原则

Prompt 应尽可能短、稳定、可复用，不让模型承担语言判断和路由职责。

建议：

- system 只定义“翻译器角色”和输出约束
- user 只传结构化字段
- 不要求模型总结、扩写或解释

### 6. 推荐 Prompt

#### system

```text
你是一个会议实时翻译器。
请将用户提供的文本准确翻译为指定目标语言。
要求：
1. 保留原句语气，不做扩写。
2. 不总结、不解释、不补充背景。
3. 专有名词、API、产品名尽量保持原样。
4. 输出仅包含译文，不要附加说明。
```

#### user

```text
source_language: zh
target_language: ja
context_tail:
- 前一条 segment
- 前两条 segment
text: 这个 API 先走 websocket，后面再补 token 逻辑。
```

### 7. 关于 `text:` 这一行是否必须保留

建议保留。

原因：

- `text:` 代表“当前真正需要被翻译的 segment 原文”
- 它和 `context_tail` 的职责不同
- `context_tail` 是辅助理解上下文
- `text` 才是本次要输出译文的目标内容

如果去掉 `text:` 这一行，而只把一句话裸放在 user prompt 里，也不是绝对不行，但会带来两个问题：

- 结构不够稳定，后续加 `context_tail` 时容易混淆
- 脚本批量测试和日志记录时，不便于程序组装和排错

因此，建议保留 `text:` 字段。

## 九、翻译执行器设计

翻译执行器是实际把原文变成译文的能力模块。

### 当前建议

把翻译执行器单独抽象出来，不直接写死在 ASR 任务里。

这样后续可以替换不同实现：

- 执行器 A：`DeepThink` 自带翻译能力
- 执行器 B：独立轻量翻译服务
- 执行器 C：LLM 翻译能力

### 当前阶段结论

翻译执行器需要进一步调研，重点关注：

- 实时性是否足够
- 是否免费或成本可控
- 是否可接商汤现有能力
- 是否支持热词库或术语约束
- 是否适合短句高频调用

### 当前临时占位策略

在正式翻译执行器未确定前，界面展示可采用临时占位方案：

- 默认指定 `DeepThink` 按 `en` 翻译进行效果占位

说明：

- 此方案仅用于当前页面展示和前端结构调试
- 不代表最终产品翻译架构
- 真正落地时，仍应切换到“识别后路由翻译”结构

## 十、为什么当前翻译模块先置空

原因不是“不做翻译”，而是“先把翻译执行能力和产品逻辑拆开”。

当前优先级应是：

1. 先确认会中转写主链路稳定
2. 再把翻译规则和页面结构设计正确
3. 最后选择合适的翻译执行器接入

这比一开始把翻译逻辑硬编码进 ASR 参数里更稳。

## 十一、MVP 阶段的推荐落地方案

### 当前建议保留的设计原则

- 原文始终优先展示
- 译文异步补充
- 翻译策略独立于 ASR
- 会议语言配置可扩展
- 后续允许热切换

### 当前建议的 MVP 版本逻辑

- ASR：使用 `DeepThink WebSocket`
- 原文：直接展示分段识别结果
- 语言判断：使用轻量规则模块
- 翻译路由：按主阅读语言 / 会议外语目标语言决策
- 翻译执行器：当前先置空，后续调研接入
- 页面展示：先使用 `DeepThink -> en` 结果进行占位演示

## 十二、当前阶段的边界说明

当前草案暂不解决以下问题：

- 多人说话人分离
- 声纹映射到真实身份
- 多外语同时作为目标语言输出
- 历史 segment 在热切换后的全量重翻
- 复杂术语库对翻译执行器的实时生效方式

这些能力可以在后续版本中逐步补齐。

## 十三、一句话总结

本翻译模块的核心思想是：

- 用 `DeepThink` 稳定拿原文
- 用语言判断模块确定原始语言
- 用翻译路由层决定该翻成什么语言
- 用可替换的翻译执行器完成译文生成
- 用页面层实现“原文优先、译文补充、支持热切换”的双语会议体验

这套方案的价值在于：

- 既保留当前 MVP 的开发速度
- 又避免后续被单一模型能力限制住产品演进空间
