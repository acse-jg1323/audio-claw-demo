# 术语存储层设计

> 版本：v0.1  
> 日期：2026-06-23  
> 目标：明确术语热词库在当前 MVP 中的存储结构、检索方式、语言对命中策略、Prompt 提取方式，以及 Excel 导入导出映射。

## 一、设计结论

这层设计建议采用“两层模型”：

- **持久化层**：存“术语条目 + 方向规则”，用于 `localStorage`、页面展示、导入导出、编辑维护
- **运行时层**：从持久化数据编译出“按语言对组织的命中索引”，用于会中快速检索

这样做的原因是：

- 原型页天然是“表格行视角”，更接近方向规则
- 实时翻译命中时，本质也是“当前源语言 + 当前目标语言”下查一组规则
- 如果直接只存概念型多语言对象，运行时还要临时拆规则，编辑层也不直观
- 如果直接只存纯方向行，又容易在导入、双向、多语言术语时失去归组能力
- 企业共享与个人术语在数据形态上其实没有本质区别，区别主要来自归属和权限

因此最合适的做法是：

- **一个术语条目 `entry` 负责归组、权限、统计、元信息**
- **一个术语条目下可以有多条 `rule`，每条 `rule` 对应一个具体语言对下的命中与输出规则**
- **个人术语和企业术语共用同一套 `entry + rule` 存储结构，不拆成两套独立 schema**

## 一点补充：企业与个人是否应当分开存储

我认为你的想法是合理的，推荐明确采用：

- **同一套存储模型**
- **同一个顶层 `entries` 数组**
- **通过 `scope + ownerUserId + companyKey + 当前登录用户权限` 区分个人与企业**

也就是说，企业共享并不是另一套“专门的企业表结构”，而是同一个术语模型中的另一种归属方式。

这样做的好处是：

- 页面层更简单：`我的术语` 和 `企业共享` 只是同一数据集下的不同过滤视图
- 编辑层更统一：新增、编辑、删除、导入、导出都走同一套 store API
- 运行时更直接：命中时只需要先组装“当前用户可用术语集合”，不需要跨两套结构做二次拼装
- 后续更好扩展：如果未来增加“团队共享”“部门共享”，也可以继续沿用同一模型

因此，企业级和个人级的差异，建议只体现在下面几个字段和规则上：

- `scope`：`personal` 或 `enterprise`
- `ownerUserId`：个人术语归属用户
- `companyKey`：企业术语归属公司
- 页面权限：`admin` 可编辑企业共享，`member` 对企业共享只读
- 运行时可见性：个人只能看到自己的个人术语，但企业共享对同公司成员都可见

## 二、设计目标

本设计需要同时满足四类需求：

### 1. 满足原型页字段

至少需要覆盖 `terms.html` 目前已经展示出来的信息：

- 源词
- 译法
- 类型
- 语言对
- 命中次数
- 维护者
- 更新时间
- 我的术语 / 企业共享

### 2. 满足开发实际需要

除了页面字段，还必须补齐：

- 主键与归组字段
- 用户 / 企业归属
- 启用 / 禁用状态
- 别名 / 缩写 / 变体
- 命中规则
- 输出模式
- 导入来源
- 创建 / 更新时间

### 3. 满足运行时检索

实时翻译命中必须能够高效回答下面这个问题：

> 当前 segment 的 `sourceLanguage` 和 `desiredTargetLanguage` 已知，文本是 `text`。  
> 哪些术语规则应当命中？命中后应向 Prompt 传哪几个映射？

### 4. 满足 Excel 导入导出

导入导出需要：

- 用户能看懂
- 一行最好就是一个明确规则
- 支持多语言、多规则归组
- 导入后能映射到内部结构

## 三、从原型页反推最小必要字段

结合 `app/terms.html` 当前原型，页面最少已经隐含了以下信息：

### 我的术语

- `sourceText`
- `targetText`
- `termType`
- `languagePair`
- `hitCount`
- 可编辑 / 可删除

### 企业共享

- `sourceText`
- `targetText`
- `maintainerName`
- `updatedAt`
- 企业共享只读或可编辑

### 原型中已经出现的术语类型特点

- 单向固定译法：如 `神泉科技 -> シェンチュアン・テクノロジー`
- 单向保留原文：如 `サンライズ -> サンライズ（保留原文）`
- 多语言条目：如 `APPI`
- 识别热词：如 `田中健太`、`POC`

因此，存储层不能只支持“简单一对一翻译”，还必须支持：

- 单向规则
- 双向规则
- 多规则归组
- 保留原文输出
- 类型区分

## 四、推荐存储模型

## 4.1 `localStorage` 存储键

建议：

```js
const STORAGE_KEY = "ac_terms_v2";
```

之所以用 `v2`，是因为相比前面粗粒度方案，这一版把“条目”与“规则”明确分开了。

## 4.2 持久化顶层结构

建议持久化成下面这种结构：

```json
{
  "version": 2,
  "entries": [],
  "updatedAt": "2026-06-23T12:00:00.000Z"
}
```

其中 `entries` 是主数组，每个元素代表一个术语条目。

## 4.3 术语条目 `entry`

建议结构：

```json
{
  "entryId": "term_ent_001",
  "conceptKey": "audio_claw",
  "scope": "enterprise",
  "ownerUserId": null,
  "companyKey": "神泉科技",
  "termType": "fixed_translation",
  "displayName": "AudioClaw",
  "notes": "产品名固定译法",
  "tags": ["产品名"],
  "status": "enabled",
  "priority": 100,
  "createdAt": "2026-06-23T12:00:00.000Z",
  "createdByUserId": "u_admin_001",
  "createdByName": "李明",
  "updatedAt": "2026-06-23T12:00:00.000Z",
  "updatedByUserId": "u_admin_001",
  "updatedByName": "李明",
  "stats": {
    "hitCount": 87,
    "lastHitAt": "2026-06-23T12:10:00.000Z"
  },
  "rules": []
}
```

### `entry` 字段说明

- `entryId`：条目主键
- `conceptKey`：逻辑归组键，用于多语言 / 多规则归组
- `scope`：`personal` 或 `enterprise`，用于区分同一套模型下的个人术语与企业术语
- `ownerUserId`：个人术语归属用户，企业术语为空
- `companyKey`：企业共享数据的归属公司，个人术语可为空
- `termType`：`fixed_translation` 或 `recognition_hotword`
- `displayName`：页面上默认展示名称
- `notes`：备注
- `tags`：可选标签，如“产品名”“人名”“法务术语”
- `status`：`enabled` / `disabled`
- `priority`：同类冲突时的人为优先级
- `stats`：条目级命中统计
- `rules`：该条目下的具体规则数组

## 4.4 方向规则 `rule`

建议每个 `entry.rules` 元素结构如下：

```json
{
  "ruleId": "rule_ent_001_zh_ja",
  "sourceLanguage": "zh",
  "targetLanguage": "ja",
  "sourceTerms": ["AudioClaw"],
  "sourceAliases": ["audio claw", "AUDIOCLAW"],
  "targetText": "オーディオクロー",
  "targetMode": "translate",
  "matchMode": "contains",
  "caseSensitive": false,
  "wholeWord": false,
  "bidirectionalGroup": "bg_audio_claw_001",
  "status": "enabled",
  "stats": {
    "hitCount": 87,
    "lastHitAt": "2026-06-23T12:10:00.000Z"
  }
}
```

### `rule` 字段说明

- `ruleId`：规则主键
- `sourceLanguage`：命中源语言，只能是 `zh` / `ja` / `en`
- `targetLanguage`：输出目标语言，只能是 `zh` / `ja` / `en`
- `sourceTerms`：主命中词数组，至少一个
- `sourceAliases`：别名、缩写、大小写变体、常见写法
- `targetText`：传给翻译 Prompt 的目标表达
- `targetMode`
  - `translate`：按 `targetText` 固定输出
  - `preserve_original`：保留原文
- `matchMode`
  - `contains`：适合中日文、术语片段
  - `whole_word`：适合英文缩写或完整单词
  - `exact`：适合特别严格的短词
- `caseSensitive`：英文场景用
- `wholeWord`：保留为运行时匹配辅助布尔值
- `bidirectionalGroup`：双向规则归组标识，便于 UI 识别“这两条是一起生成的”
- `status`：规则是否启用
- `stats`：规则级命中统计

## 五、为什么不直接只存“多语言 forms”

虽然“按语言维护 forms”看起来很优雅，但对当前项目来说不够直接，原因有三点：

### 1. 运行时命中天然是“方向性”的

当前翻译链路先有：

- `sourceLanguage`
- `desiredTargetLanguage`

然后才决定送什么 Prompt。

所以最自然的问题是：

> 在 `zh -> ja` 下，有哪些规则可用？

这个问题用方向规则结构最直接。

### 2. “保留原文”本身也是方向性的

比如：

- `サンライズ -> 保留原文`

它不是一个简单的“概念翻译”，而是一个明确的目标输出策略。

### 3. Excel 一行就是一条规则

这对导入导出和产品运营最友好。

结论：

- **条目负责归组**
- **规则负责命中与输出**

这是当前 MVP 最稳的方案。

## 六、运行时检索设计

## 6.1 核心思路

持久化数据存 `entries + rules`，但运行时不直接每次从原始结构全量临时解析，而是先编译成内存索引。

建议在 `terms-store.js` 或 `meeting-terms-matcher.js` 中编译出下面这样的结构：

```js
{
  fixedTranslationByPair: {
    "zh->ja": [compiledRule1, compiledRule2],
    "ja->zh": [compiledRule3],
    "zh->en": []
  },
  hotwordByLanguage: {
    zh: [compiledRuleA],
    ja: [compiledRuleB],
    en: [compiledRuleC]
  }
}
```

注意：

- **索引是运行时派生数据，不需要持久化进 `localStorage`**
- 页面刷新后重新从原始存储编译即可

## 6.2 术语匹配范围

### 固定译法

对当前 segment 的 Prompt 注入，只取：

- `status = enabled`
- `termType = fixed_translation`
- `sourceLanguage = 当前 segment 语言`
- `targetLanguage = 当前翻译目标语言`
- `scope ∈ {enterprise, personal}`

并按优先级合并当前用户可见的规则。

### 识别热词

本轮不作为翻译 Prompt 的主输入来源，但存储层应支持按语言读取：

- `status = enabled`
- `termType = recognition_hotword`
- `sourceLanguage = 当前 segment 语言`

它后续可用于：

- ASR 热词增强
- ASR 后置标准化
- 术语页统计与管理

## 6.3 当前用户下的规则来源

运行时应先拼出“当前用户可用规则集合”：

1. 企业共享术语：`scope = enterprise && companyKey = currentUser.company`
2. 我的术语：`scope = personal && ownerUserId = currentUser.id`

这里需要强调：

- 这是**同一套存储中的两类可见数据**
- 不是先从“企业库”再从“个人库”跨结构读取
- 更准确地说，是在同一个 `entries` 数据集中按归属与权限过滤出“当前用户可用规则集合”

然后做统一命中。

## 6.4 检索优先级

建议命中冲突时按以下顺序排序：

1. `enterprise` 高于 `personal`
2. `priority` 高于默认优先级
3. `sourceTerms` 中更长的词优先
4. 更晚更新的规则优先

这样能避免：

- 短词把长词提前抢走
- 个人术语覆盖企业统一译法

## 七、搜什么字段，怎么检索

这是这层设计最关键的问题。

## 7.1 检索输入

每次会中检索的输入是：

```js
{
  text,
  sourceLanguage,
  targetLanguage,
  currentUser
}
```

其中：

- `text` 来自当前 segment
- `sourceLanguage` 来自 `detectLanguage()`
- `targetLanguage` 来自 `createTranslationPlan()`

## 7.2 实际参与检索的字段

运行时主要搜下面几个字段：

- `rule.sourceLanguage`
- `rule.targetLanguage`
- `rule.sourceTerms`
- `rule.sourceAliases`
- `rule.status`
- `entry.scope`
- `entry.ownerUserId`
- `entry.companyKey`
- `entry.termType`
- `entry.priority`

其中真正做文本匹配的字段只有：

- `sourceTerms`
- `sourceAliases`

其他字段都是“缩小候选集”的过滤条件。

## 7.3 为什么不搜 `targetText`

因为在会中翻译前，系统拿到的是**原文 segment**，不是译文。

所以 Prompt 注入前的命中检索，应该永远围绕：

- 当前原文长什么样
- 当前原文可能有哪些别名写法

进行，而不是围绕目标译文。

`targetText` 的职责是：

- 命中后被提取出来
- 作为术语映射传给 Prompt

## 7.4 语言差异下的匹配建议

### 中文 / 日文

建议默认：

- `matchMode = contains`

原因：

- 中日文没有天然空格分词
- 当前 segment 通常较短
- 术语量不大时，直接 `includes` 已足够

### 英文

建议默认：

- `matchMode = whole_word`
- `caseSensitive = false`

原因：

- 英文短词、缩写较多
- 直接 `includes` 容易误伤，比如 `api` 命中 `capital`

### 超短词

像：

- `AI`
- `PI`
- `POC`

建议允许配置：

- `matchMode = exact` 或 `whole_word`

不要默认走宽松匹配。

## 八、怎么提取并传入 Prompt

## 8.1 提取结果格式

命中后，不需要把整个条目都传给大模型，只要抽取当前语言对下真正有效的映射即可。

建议匹配函数输出：

```js
[
  {
    entryId: "term_ent_001",
    ruleId: "rule_ent_001_zh_ja",
    sourceText: "神泉科技",
    targetText: "シェンチュアン・テクノロジー",
    targetMode: "translate",
    scope: "enterprise",
    priority: 100
  }
]
```

## 8.2 Prompt 传输格式

建议在 `buildUserPrompt()` 中增加：

```text
glossary_terms:
- 神泉科技 => シェンチュアン・テクノロジー
- AudioClaw => オーディオクロー
```

对于 `preserve_original`，建议传成：

```text
glossary_terms:
- サンライズ => 保留原文（输出时保留 source text）
```

也可以在程序层直接把 `targetText` 处理成源词本身，再传：

```text
- サンライズ => サンライズ
```

在 MVP 阶段，后者更简单稳定。

## 8.3 Prompt 传输上限

每条 segment 不建议传太多术语。

建议：

- 最多传 3 到 5 条命中规则

选取顺序：

- 企业优先
- 长词优先
- 高优先级优先

## 九、页面展示字段与存储字段映射

| 页面字段 | 来源 |
| --- | --- |
| 源词 | `rule.sourceTerms[0]` |
| 译法 | `rule.targetText` 或“保留原文” |
| 类型 | `entry.termType` |
| 语言对 | `rule.sourceLanguage + rule.targetLanguage` |
| 命中 | `rule.stats.hitCount` 或 `entry.stats.hitCount` |
| 维护者 | `entry.updatedByName` |
| 更新时间 | `entry.updatedAt` |
| 企业 / 我的 | `entry.scope` |

说明：

- 表格按“规则行”展示最自然
- 编辑时仍然可以在弹窗里看到同组 `entry` 下的其他规则

## 十、Excel 导入导出设计

## 10.1 推荐 Excel 行模型

Excel 建议一行对应一条规则，字段如下：

| 字段名 | 是否必填 | 说明 |
| --- | --- | --- |
| `scope` | 是 | `personal` / `enterprise` |
| `term_type` | 是 | `fixed_translation` / `recognition_hotword` |
| `concept_key` | 否 | 同组规则可共用；为空时导入时自动生成 |
| `display_name` | 否 | 页面默认展示名 |
| `source_language` | 是 | `zh` / `ja` / `en` |
| `source_text` | 是 | 主命中词 |
| `source_aliases` | 否 | 多个值可用 `；` 或 `|` 分隔 |
| `target_language` | 是 | `zh` / `ja` / `en` |
| `target_text` | 否 | 当 `target_mode = preserve_original` 时可为空 |
| `target_mode` | 是 | `translate` / `preserve_original` |
| `match_mode` | 否 | `contains` / `whole_word` / `exact` |
| `priority` | 否 | 数字，默认 100 |
| `notes` | 否 | 备注 |

## 10.3 导入后的内部归组

导入时：

- 相同 `concept_key`
- 相同 `scope`
- 相同归属用户或公司

可以归并为同一个 `entry`。

如果 `concept_key` 为空，则：

- 自动按 `display_name + scope + owner/company` 生成一个稳定键

## 十一、命中统计建议

建议同时保留两层统计：

### 条目级统计

用于页面卡片、整体排序：

- `entry.stats.hitCount`
- `entry.stats.lastHitAt`

### 规则级统计

用于精确反映某个语言对命中情况：

- `rule.stats.hitCount`
- `rule.stats.lastHitAt`

因为同一个术语在：

- `zh -> ja`
- `en -> zh`

的命中频率可能完全不同。

## 十二、我当前最推荐的最终方案

如果只给一个结论，我最推荐的是：

### 持久化

- 顶层：`entries`
- 条目下：`rules`

### 页面

- 列表按 `rule` 行展示
- 编辑按 `entry` 归组处理

### 运行时

- 用 `sourceLanguage + targetLanguage` 编译成 pair index
- 文本命中只搜 `sourceTerms + sourceAliases`
- Prompt 只提取当前命中的有效映射

### 类型策略

- `fixed_translation`：本轮真实参与翻译 Prompt
- `recognition_hotword`：本轮完成存储与管理，后续再接 ASR 热词增强

## 十三、需要你确认的几个点

我建议你重点确认下面 4 个决策：

1. 是否接受“**条目归组 + 规则行命中**”这套结构，而不是只存纯概念对象
2. Excel 是否接受“一行一条规则”的导入模型
3. `preserve_original` 是否按“导入时可不填 `target_text`”处理
4. 本轮是否确认 `recognition_hotword` 只先做资产层，不强行接入 ASR 实时热词参数
