# 代码库结构说明

## 一、文档目的

本文档用于说明当前 MVP 阶段的主要代码库结构，帮助后续开发时统一文件摆放方式，避免真实页面、原型页面、业务逻辑和文档再次混杂。

当前阶段以“够用、清晰、便于继续开发”为原则，不做过度拆分。

---

## 二、当前主结构

```text
项目根目录/
├── app/                 # 真实页面入口
├── src/                 # 真实功能代码
│   ├── integrations/    # 外部能力接入
│   └── meeting/         # 会议主链路逻辑
├── docs/                # 产品与开发文档
├── 原型/                # 原型存档
├── scripts/             # 独立脚本
├── asr-websocket-validate.html
├── senseaudio-test.html
└── vercel.json
```

---

## 三、各目录含义

### 1. `app/`

这里放当前真实可访问、可继续开发的产品页面。

当前包括：

- `index.html`：会议工作区
- `home.html`：主页
- `terms.html`：术语页
- `voiceprint.html`：声纹页
- `settings.html`：设置页
- `admin.html`：管理后台页

约定：

- `app/` 中只放真实页面
- 后续新增 `login.html` 也应放在这里
- 页面一旦进入真实开发，就不要继续只留在 `原型/`

### 2. `src/`

这里放真实功能代码，不放产品文档，不放原型页面。

当前分为两块：

#### `src/integrations/`

用于存放外部能力接入相关代码。

当前包括：

- `audio-pcm-capture.js`：音频采集
- `senseaudio-websocket-client.js`：ASR WebSocket 接入
- `asr-websocket-validate.js`：ASR 接入验证逻辑

#### `src/meeting/`

用于存放会议主链路相关逻辑。

当前包括：

- `meeting-page-app.js`：会议页主逻辑
- `meeting-summary-executor.js`：会议总结生成逻辑
- `meeting-translation-executor.js`：翻译执行逻辑
- `meeting-translation-router.js`：翻译路由逻辑

约定：

- 会议页主逻辑优先放在 `src/meeting/`
- 与第三方服务、音频采集、ASR 通信更相关的逻辑放在 `src/integrations/`

### 3. `docs/`

这里放所有产品、设计、开发说明类文档。

当前包括：

- 系统功能梳理
- MVP 任务分解
- 会议总结设计
- ASR 接入说明
- Home-会议闭环方案
- 翻译测试记录

约定：

- 根目录不再继续散放新的说明文档
- 新的方案、设计、确认稿优先放入 `docs/`

### 4. `原型/`

这里放原始高保真原型文件，仅作为设计档案和开发参考。

约定：

- `原型/` 中的页面不作为真实运行页面
- 真实开发以 `app/` 中页面为准
- 原型保留，不删，用于对照视觉和交互

### 5. `scripts/`

这里放独立运行的辅助脚本。

当前包括：

- `translation-model-benchmark.mjs`：翻译模型测试脚本

### 6. 根目录其他文件

- `asr-websocket-validate.html`：ASR 验证页
- `senseaudio-test.html`：测试页
- `vercel.json`：部署配置
- `.vercel/`：Vercel 相关配置

当前理解：

- 这类文件暂时保留在根目录即可
- 它们不属于产品主页面
- 后续如数量增多，再单独整理

---

## 四、当前开发约定

为保证后续开发持续整齐，当前建议统一遵循以下约定：

1. 真实页面只放在 `app/`
2. 原型页面只放在 `原型/`
3. 页面逻辑放在 `src/`
4. 外部接入代码放在 `src/integrations/`
5. 会议主链路代码放在 `src/meeting/`
6. 设计与方案文档放在 `docs/`

---

## 五、当前阶段说明

当前代码库已经完成了最基本的分层：

- 页面层：`app/`
- 逻辑层：`src/`
- 文档层：`docs/`
- 原型层：`原型/`

后续随着 `login`、本地数据存储、权限控制等功能继续开发，再逐步补充新的目录，但当前阶段不需要继续过度细分。
