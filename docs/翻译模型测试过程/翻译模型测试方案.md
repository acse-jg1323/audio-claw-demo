# 翻译模型测试方案

> 适用阶段：智能会议系统 MVP
> 当前目标：为“识别后路由翻译”选择一款适合作为会中逐 segment 翻译执行器的商汤文本模型
> 本轮范围：优先测试商汤自有模型 `senseaudio-s2-flash`、`senseaudio-s2-lite`、`senseaudio-s2`

## 一、测试目标

本轮测试重点不是比较大模型的综合能力，而是比较它们是否适合“会议场景中的逐 segment 文本翻译”。

因此，测试目标聚焦以下问题：

- 哪个模型延迟更低
- 哪个模型对短句、口语句的翻译更稳定
- 哪个模型对 API、产品名、术语的保留更自然
- 哪个模型更适合作为会中近实时翻译执行器

## 二、测试模型

本轮测试固定以下三款模型：

- `senseaudio-s2-flash`
- `senseaudio-s2-lite`
- `senseaudio-s2`

角色预期：

- `senseaudio-s2-flash`：会中实时翻译优先候选
- `senseaudio-s2-lite`：低成本候选
- `senseaudio-s2`：质量上限参考

## 三、测试接口

使用 SenseAudio Token Plan 的 OpenAI 兼容接口：

- Base URL：`https://api.senseaudio.cn/v1`
- Endpoint：`/chat/completions`

鉴权方式：

- `Authorization: Bearer $SENSEAUDIO_API_KEY`

## 四、固定 Prompt

### system

```text
你是一个会议实时翻译器。
请将用户提供的文本准确翻译为指定目标语言。
要求：
1. 保留原句语气，不做扩写。
2. 不总结、不解释、不补充背景。
3. 专有名词、API、产品名尽量保持原样。
4. 输出仅包含译文，不要附加说明。
```

### user 结构

```text
source_language: zh
target_language: ja
context_tail:
- 前一条 segment
- 前两条 segment
text: 这个 API 先走 websocket，后面再补 token 逻辑。
```

说明：

- `source_language`：由规则语言判别模块给出
- `target_language`：由翻译路由层给出
- `context_tail`：可选，默认传最近 `1-2` 条 segment
- `text`：当前真正需要翻译的原文

## 五、统一测试样本

本轮先使用 9 条测试样本，覆盖中日英、技术术语、口语省略和上下文承接场景。

### 1. 中文 -> 日语

```text
source_language: zh
target_language: ja
context_tail:
- 我们先把会议页跑起来。
text: 这个 API 先走 websocket，后面再补 token 的鉴权层。
```

### 2. 中文 -> 英文

```text
source_language: zh
target_language: en
context_tail:
- 现在先不要改总结模块。
text: 这一版先保证原文稳定展示，译文异步补上就可以。
```

### 3. 中文口语 -> 日语

```text
source_language: zh
target_language: ja
context_tail:
- 这个地方还没完全定。
text: 对，就先这么搞，后面我们再慢慢收。
```

### 4. 日语 -> 中文

```text
source_language: ja
target_language: zh
context_tail:
- 先把 MVP 的核心链路跑通。
text: まず会議中のリアルタイム翻訳だけ安定させれば十分です。
```

### 5. 日语商务句 -> 中文

```text
source_language: ja
target_language: zh
context_tail:
- 现在讨论的是术语处理策略。
text: 専門用語については、企業ごとの用語集をあとから差し込めるようにしたいです。
```

### 6. 英文 -> 中文

```text
source_language: en
target_language: zh
context_tail:
- We already finished the websocket validation page.
text: We should keep the original transcript first and append the translated segment asynchronously.
```

### 7. 英文技术句 -> 中文

```text
source_language: en
target_language: zh
context_tail:
- The ASR result comes from DeepThink.
text: Please preserve product names, API terms, and websocket related wording in the translated output.
```

### 8. 中文夹英文术语 -> 日语

```text
source_language: zh
target_language: ja
context_tail:
- 这里要跟前端联动一下。
text: 这条消息先不要走 summary，只走 segment 翻译和 UI append。
```

### 9. 日文夹英文产品名 -> 中文

```text
source_language: ja
target_language: zh
context_tail:
- 今は会中翻訳だけを見ています。
text: AudioClaw の meeting page は、まず original transcript を先に表示したいです。
```

## 六、测试指标

每个模型都按以下 5 个维度记录。

### 1. 延迟

- 单条请求从发出到收到回复的耗时
- 记录单位：`ms`

### 2. 稳定性

- 输出是否总是只返回译文
- 是否有多余解释、前缀、总结、换行污染

### 3. 术语表现

- 是否能保留 `API`、`websocket`、`token`、`AudioClaw` 等术语
- 是否乱翻或过度本地化

### 4. 口语适应

- 对口语、省略、短句承接是否自然
- 是否出现漏译、过度书面化

### 5. 上下文利用

- 使用 `context_tail` 后是否更好理解当前句
- 是否会错误地把历史内容翻进当前句

## 七、执行方式

建议使用一份本地测试脚本统一执行三模型请求。

脚本要求：

- 统一使用同一套 prompt
- 统一使用同一套样本
- 自动记录每次调用耗时
- 输出每个模型、每条样本的翻译结果
- 最后给出便于人工比较的汇总结果

## 八、通过标准

如果某款模型同时满足以下条件，则可以优先作为 MVP 翻译执行器候选：

- 平均延迟明显低于旗舰模型
- 大多数输出仅包含译文，无额外解释
- 术语保留稳定
- 中文 / 日语 / 英文的基础翻译表现可接受
- 在口语句上没有明显大面积漏译

## 九、预期结论形式

本轮测试结束后，应输出一份结论，至少包括：

- 三款模型的平均延迟对比
- 三款模型的主观质量判断
- 推荐哪一款用于 MVP 会中翻译
- 哪一款适合作为备选
- 哪一款更适合作为高质量参考而非默认链路
