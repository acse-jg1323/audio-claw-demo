# 术语热词库模块开发方案

> 版本：v0.3\
> 日期：2026-06-23\
> 目标：基于当前智能会议 MVP 的现有前端架构，落地“术语热词库”的真实可用能力，包括本地存储、权限控制、Excel 导入导出、页面编辑，以及会中翻译链路联动。

## 最新迭代口径

以下内容是本轮开发过程中，针对 MVP 产品复杂度、用户理解成本和运行时稳定性做出的**最新口径修订**。后续 Excel 导入导出、术语匹配和翻译接入，统一以这部分为准。

### 1. 产品侧降低用户负担

- 不要求普通用户手动理解或配置“优先级”
- 不要求普通用户手动选择 `contains / whole_word / exact` 等匹配模式
- 页面与 Excel 模板都应优先呈现业务字段，而不是程序内部字段

这意味着：

- 用户主要只需要关心“术语是什么、适用于哪个语言对、是固定翻译还是保留原文”
- 系统内部仍可保留必要的匹配与裁决逻辑，但 MVP 首版不把这些概念暴露给用户

### 2. 同一术语、同一语言对，只允许一条规则

为避免用户录入自相矛盾的规则，MVP 口径调整为：

- **同一个术语条目**
- **在同一个源语言 -> 目标语言的映射下**
- **只能存在一条规则**

并且这条规则只能二选一：

- 固定翻译
- 保留原文

不能同时存在两条互相打架的规则，例如：

- 同一个术语在 `ja -> zh` 下同时配置“翻译成某个中文词”
- 又配置“保留原文”

这类情况在 MVP 录入与导入阶段应直接禁止。

### 3. 企业共享优先于个人术语

运行时如果同一个 segment 同时命中：

- 企业共享术语
- 个人术语

则默认：

- **企业共享优先**
- 个人术语只作为企业术语未覆盖时的补充

这条规则是为了保持企业统一口径，避免企业已经维护好的术语在实际翻译中被个人偏好覆盖。

### 4. 匹配模式保留在系统内部，但不暴露给用户

MVP 首版不建议让用户思考匹配策略，但系统内部仍保留默认匹配逻辑：

- 中文 / 日文：默认按“句子里包含该术语片段”处理，即 `contains`
- 英文：默认按“作为完整单词出现”处理，即 `whole_word`

产品表达上可以直接理解为：

- 中文 / 日文：一句话里只要出现该术语，就算命中
- 英文：只有在句子里作为完整单词出现时才算命中，避免缩写误伤

### 5. Excel 模板要固定、简单、面向业务

后续 Excel 导入模板应遵循：

- 固定表头
- 一行表示一条规则
- 不暴露程序内部字段，如 `entryId`、`ruleId`、`conceptKey`
- 不要求用户填写优先级和匹配模式

用户应填写的字段，优先保持为他们自然能理解的内容，例如：

- 术语名称
- 归属范围
- 术语类型
- 源语言
- 目标语言
- 源词
- 别名
- 处理方式
- 译法
- 备注

### 6. 当前 MVP 修改逻辑总结

这次你对产品逻辑的质疑和修订，我认为是合理且必要的，核心变化可以总结为：

- 从“给用户更多自由配置能力”收敛为“系统内部兜底，用户侧尽量极简”
- 从“允许复杂规则组合”收敛为“同术语同语言对只保留一种明确处理方式”
- 从“潜在优先级配置”收敛为“企业优先、系统自动裁决、用户不感知”
- 从“让用户理解命中策略”收敛为“用户只关心术语业务信息，匹配策略由系统自动处理”

后续第五步和第六步都应按这套 MVP 收敛逻辑继续实现。

## 一、结论先行

基于当前代码库现状，这个模块**完全可做**，而且适合按 MVP 方式优先落在前端本地存储与会中翻译联动上。

你的核心想法总体是对的：

- 先按 segment 落文后的 `text` 做术语检索
- 再把命中的术语条目拼进翻译 prompt
- 没命中时保持当前翻译链路不变
- 命中时只给 prompt 增加少量术语约束，不额外引入新的模型能力

这条路线和当前仓库的真实架构是兼容的，因为目前项目本身就是：

- 前端页面为主
- `localStorage` 负责 MVP 数据持久化
- 会议实时链路在前端完成语言判断与翻译路由
- 翻译执行器已经预留了 `glossary_terms` 概念，但尚未落地

因此，本轮建议把“术语热词库”拆成两个层次：

- **P0 必做**：真实术语存储、页面 CRUD、Admin/Member 权限、Excel 导入导出、固定译法接入翻译 prompt
- **P1 延展**：识别热词真正参与 ASR 热词增强，或做更复杂的命中统计、跨页面管理后台复用、跨标签实时同步

`app/terms.html` 当前还是高保真原型页，已经具备：

也就是说，它的 UI 基本够用，但还没有真正的数据源、导入导出、权限控制与编辑逻辑。

### 2. 现有权限基线

项目当前已经有最小账号体系：

- `admin`：李明
- `member`：王敏

并且已通过 `src/auth/auth-store.js` 与 `src/auth/page-auth.js` 实现：

- 管理员页面拦截
- 普通成员隐藏“组织管理”导航

文档口径也已经明确：

所以术语页的权限并不需要重造，只需要在现有登录态基础上补齐页面级编辑限制即可。

### 3. 现有翻译链路

当前实时会议链路已经明确为：

1. WebSocket ASR 返回原始 segment
2. 前端 `detectLanguage()` 判断语种
3. `createTranslationPlan()` 决定目标语言
4. `translateSegment()` 调用 `senseaudio-s2-lite`
5. 译文异步回填页面

这意味着“术语热词库”的最佳接入点，并不是 ASR 前，而是：

- **在 segment 已形成文本后**
- **在** **`requestTranslation()`** **调用翻译执行器之前**

这恰好和你的方案一致。

## 三、对你方案的可行性评估

## 3.1 可行的部分

你的方案中，下面这些点是非常适合当前 MVP 的：

- 用 `localStorage` 存术语，而不是先上数据库
- 按 `zh / ja / en` 做语言标签
- 在 segment 文本稳定后做快速术语检索
- 命中后把术语作为 prompt 约束传给翻译模型
- 未命中则走原有快速翻译路径

原因是：

- 当前项目全链路已经是前端本地模式，继续沿用最稳
- 术语量在 MVP 阶段通常是几十到几百条，全量扫描完全够快
- 每条 segment 文本较短，匹配成本非常低
- 现有翻译 prompt 本身就是字符串拼接型，增加术语块实现成本低

## 3.2 需要修正的部分

### A. “固定译法”和“识别热词”不能完全等同

当前原型页里有两种类型：

- 固定译法
- 识别热词

这两类在当前 MVP 中的理解建议统一为：

- **固定译法**：重点解决“这个词最终应该怎么翻译”。更偏翻译输出约束，适合企业统一术语、产品名、人名、公司名等需要固定译法的场景。
- **识别热词**：重点解决“这个词应该被识别出来、保留下来、重点关注”。更偏识别资产与热词资产，适合领域热词、专有名词、缩写、人名等需要重点命中的词。

需要特别说明的是：

- 两者在当前实现里**共用同一套存储结构和编辑方式**，背后都是同一个术语资产模型，只是 `termType` 不同。
- 在当前 MVP 阶段，两者首先体现为**页面上的两种术语分类**，支持筛选、统计、导入导出和管理。
- 真正的能力区别在运行链路里：
  - **固定译法**优先服务翻译环节，用于把命中的术语传入 prompt，约束译文输出。
  - **识别热词**更适合未来接入 ASR 热词增强或识别后标准化，本轮先按资产类型管理，不作为核心交付承诺。

但你现在提出的接入方式，本质上主要解决的是**翻译约束**，即“某个词应该怎么翻译”。

这对于“固定译法”非常合适，但对“识别热词”并不完全等价。因为“识别热词”严格来说更接近：

- ASR 前置热词增强
- 或 ASR 结果的后置纠错/标准化

而当前仓库里的实时主链路使用的是 `senseaudio-asr-deepthink-1.5-260319` WebSocket，代码中尚未存在热词参数的接法，也没有现成的热词注入实现。

因此建议本轮做法如下：

- **固定译法**：本轮真实接入翻译链路，作为核心交付
- **识别热词**：本轮先完成存储、展示、导入导出、命中统计字段预留
- **识别热词的真实 ASR 生效**：放到下一阶段，待确认接口是否支持热词参数后再接

如果希望本轮“识别热词”也有一定真实效果，可以采用一个保守版 MVP 方案：

- 在 ASR 返回文本后，做一层轻量的后置标准化匹配
- 只对明确的、低风险的专名做替换或保留原文
- 不做激进纠错，避免误伤正常文本

但从稳妥性看，本轮不建议把这块作为主任务承诺。

### B. 不建议把内部存储直接做成“方向记录堆叠”

如果直接把每条术语存成：

- `zh -> ja`
- `ja -> zh`
- `zh -> en`
- `en -> zh`

虽然容易理解，但会带来三个问题：

- 数据重复
- 编辑一次要同步多条记录
- 多语言条目容易失配

更适合当前项目的方式是：

- **Excel 导入导出可以按方向行表达**
- **内部存储用“概念条目”统一管理多语言文本**

这样既方便用户编辑，也方便程序做双向匹配。

## 4.2 Excel 导入导出模型

虽然内部建议用“概念条目”存储，但 Excel 交互层建议用更直观的“方向行”格式。

推荐模板字段：

- `source_language`
- `source_text`
- `target_language`
- `target_text`
- `term_type`
- `scope`
- `bidirectional`
- `aliases`
- `notes`

示例：

| source\_language | source\_text | target\_language | target\_text   | term\_type         | scope      | bidirectional | aliases | notes |
| ---------------- | ------------ | ---------------- | -------------- | ------------------ | ---------- | ------------- | ------- | ----- |
| zh               | 神泉科技         | ja               | シェンチュアン・テクノロジー | fixed\_translation | enterprise | true          | 神泉      | 公司名   |
| ja               | オンプレミス       | zh               | 私有化部署          | fixed\_translation | personal   | false         | <br />  | 销售常用  |

导入时流程建议为：

1. 读取 Excel 行
2. 按 `source/target language` 解析
3. 归并到内部 `forms` 结构
4. 做字段校验与去重
5. 生成导入报告

导出时则反向展开成方向行，便于业务方继续编辑。

## 五、权限与同步设计

## 5.1 页面权限规则

在 `terms.html` 上，建议直接按当前登录用户角色控制：

- `member`
  - 可增删改“我的术语”
  - 可查看“企业共享”

## 5.2 Admin 修改传导到 Member 的方式

在当前 MVP 约束下，建议这样实现：

- 企业术语统一写入 `enterpriseByCompany[currentUser.company]`
- Member 进入 `terms.html` 时按自己的 `company` 读取同一份企业数据

由于当前 demo 用户都在同一个浏览器里走 `localStorage`，所以：

- 李明修改企业术语后
- 王敏重新登录、刷新页面，或重新进入术语页
- 就能看到最新企业共享结果

这是**当前 MVP 可接受的“同步”定义**。

需要明确限制：

- 它只在同一浏览器环境内成立
- 不具备跨设备、跨浏览器同步能力
- 如果后续要做真实多端同步，只需要把 `terms-store.js` 的存储适配层从 `localStorage` 换成服务端接口即可

## 5.3 推荐优先级规则

当个人术语与企业术语冲突时，建议优先级如下：

1. 企业共享
2. 个人术语
3. 长词优先于短词
4. 同长度时按最近更新时间

原因是企业术语本身承担“组织统一译法”的职责，尤其符合你现在希望“admin 改完，member 跟着用”的诉求。

## 六、翻译链路接入方案

## 6.1 最佳接入位置

建议新增一个术语匹配模块，例如：

```text
src/meeting/meeting-terms-matcher.js
```

最佳接入位置有两个可选点：

- **方案 A，推荐**：在 `createSegmentRecord()` 阶段生成 `glossaryMatches`
- **方案 B**：在 `requestTranslation()` 里即时查找

更推荐方案 A，因为它的好处是：

- 命中结果能挂在当前 `record` 上
- 后续便于展示“本条命中术语”
- 便于累积 hit 统计
- 翻译执行器保持纯粹，只接收已准备好的参数

推荐流程：

1. ASR 返回 `text`
2. `detectLanguage(text)` 得到 `sourceLanguage`
3. `createTranslationPlan()` 得到 `targetLanguage`
4. `resolveGlossaryMatches({ text, sourceLanguage, targetLanguage, user })`
5. 把命中术语挂到 `record.glossaryMatches`
6. `requestTranslation(record)` 时将其传给 `translateSegment()`

## 6.2 匹配逻辑建议

MVP 阶段不需要复杂搜索引擎，建议采用：

- 先按 `sourceLanguage` 缩小范围
- 对当前语言下所有候选词做规范化匹配
- 中文/日文采用 `includes`
- 英文采用不区分大小写 + 单词边界匹配
- 命中结果按“企业优先、长词优先”排序

规范化建议：

- 去除前后空格
- 英文转小写
- 合并重复空白
- 保留原始字符，不做激进清洗

由于 segment 很短、术语量不大，这种全量扫描足够快。

## 6.3 Prompt 注入方式

建议在当前翻译 prompt 结构中增加一段术语块，而不是把术语描述散落到自然语言长文里。

推荐增加后的 `user prompt` 结构如下：

```text
source_language: zh
target_language: ja
context_tail:
- 我们计划下周上线企业控制台。
- 这次会议主要讨论 AudioClaw 的日文版本。
glossary_terms:
- AudioClaw => オーディオクロー
- 企业控制台 => 企業コンソール
text: AudioClaw 的企业控制台需要先做权限拆分。
```

同时建议把 `system prompt` 调整为：

- 若 `glossary_terms` 中存在明确映射，优先采用指定译法
- 但若上下文明显不是该含义，可自然翻译
- 输出只包含译文

这样既保留你说的“术语优先”，也避免极端误译。

## 6.4 对当前代码的最小改动点

建议改动以下位置：

  - 在生成 `record` 后补术语命中
  - 调用翻译时把 `glossaryMatches` 传入

  - 扩展 `translateSegment()` 参数

## 九、Excel 导入导出实现建议

当前仓库没有现成的 Excel 解析依赖，也没有 npm 工程结构。\
因此推荐两种实现路径：

### 方案 A，推荐

- CSV 导出
- 页面文案仍写“导入表格”

## 十二、推荐开发顺序

建议实际落地按下面顺序推进：

1. 先做 `terms-store.js`，把数据模型定住
2. 再做 `terms.html` 动态渲染与权限控制
3. 然后做新增/编辑/删除
4. 再做 Excel 导入导出
5. 最后接会议翻译链路

这样安排的原因是：

- 页面能先跑通真实数据闭环
- 术语格式先稳定，后面翻译链路接入不会返工
- Excel 导入导出也能复用同一套 store 结构

## 十四、风险与注意事项

### 1. `localStorage` 不是“真同步”

它只适合当前 MVP 演示和单浏览器账号切换，不适合真实企业多成员协同。

### 2. 识别热词不要在本轮过度承诺

当前最稳的真实价值来自“固定译法锁定翻译结果”，不是“ASR 热词增强已全部打通”。

### 3. 术语命中规则必须保守

尤其是英文缩写、人名、短词，过度替换会伤害翻译质量。建议：

- 长词优先
- 英文词边界匹配
- 重叠命中去重

### 4. Prompt 长度要有上限

不建议把所有命中都塞进去。建议每条 segment 最多传：

- 前 3 到 5 条最相关术语

避免 prompt 膨胀带来时延和误导。

## 十五、最终建议

综合来看，本模块最合理的落地结论是：

- **你的总体方案可行**
- **当前最值得先做的是“固定译法 + 本地存储 + 权限 + Excel 导入导出 + 翻译链路接入”**
- **“识别热词”本轮先完成资产层与界面层，ASR 真实增强作为下一阶段**

这样做既符合当前代码库能力边界，也能在演示中形成清晰价值：

- Admin 维护企业共享术语
- Member 自动继承企业统一译法
- 会中翻译真实受术语库影响
- 页面层和数据层都不是纯原型，而是真功能

这会是一个非常扎实、适合当前分支继续推进的实现路径。
