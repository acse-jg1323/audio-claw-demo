# 长会议性能优化方案（备份）

## 1. 背景

当前会议页在长会议场景下，性能压力主要集中在三个点：

- `persistTranscriptSnapshot()` 过于频繁，每次小更新都触发一次整场 transcript 快照写盘。
- 翻译、声纹识别会带来额外的重复整量写盘，中间态也会反复重写本地存储。
- 实时转写区的气泡节点持续累积，长时间会议后 DOM 数量过大，低性能设备容易卡顿甚至假死。

这三个问题叠加后，会让长会议出现明显卡顿、转写刷新延迟、页面停滞，严重时会接近浏览器本地存储与主线程承载上限。

## 2. 当前根因

### 2.1 同步整量写盘过多

- 当前 `meeting-store` 的会议数据最终写入 `localStorage`。
- `localStorage.setItem()` 是同步操作。
- 每次写入 `ac_meetings` 都会重新序列化整条会议记录，transcript 越长，单次写入越重。
- 如果每条句子、每次翻译中间态、每次声纹回填都触发写盘，主线程会被频繁阻塞。

### 2.2 翻译 / 声纹的中间态放大了写盘次数

- 句子首次到达时已经需要落一次 transcript。
- 但当前还有多处“补充信息更新”也会再次整量写盘：
  - 翻译 pending / glossary 补充
  - 翻译 success
  - 翻译 error
  - 声纹识别回填
- 其中最不值得写盘的是 streaming delta 和 pending 中间态，因为它们变化频繁、业务价值低、对最终归档影响也最小。

### 2.3 实时转写区 DOM 无上限增长

- 现在所有历史气泡会一直保留在 DOM 中。
- 长会议后，浏览器需要持续维护大量节点、样式计算、布局与绘制。
- 即便 transcript 数据本身还在，DOM 体量过大也会让页面越来越卡。

## 3. P0 目标

本轮只做最值得先做、且风险最低的快修：

1. 把实时持久化从“每次都全量写”改成“节流写”。
2. 不在每个翻译 / 声纹中间状态都重写整场 transcript。
3. 实时转写区只渲染最近一段，旧内容按需懒加载。

## 4. P0 落地口径

### 4.1 transcript 快照改为节流写

目标口径：

- 句子进入内存后，不再立刻每次都同步写盘。
- 改成按节流窗口批量落盘。
- 同时保留“强制最终快照”能力。

建议实现：

- 节流窗口：`2~5s`
- 或按条数：`5~10 条 segment`
- 任一条件命中即可执行一次批量快照
- 页面关闭、会议结束、准备进入总结 / 归档前，强制写最终快照

收益：

- 显著减少 `localStorage` 同步写次数
- 降低主线程阻塞频率
- 延长长会议可持续运行时间

### 4.2 翻译 / 声纹只在最终业务态写盘

目标口径：

- 句子首次到达时落一次
- 翻译完成后可选再落一次
- 翻译失败后可选落一次
- 声纹识别最终收口后可选落一次
- streaming delta 不落盘
- pending 中间态不落盘

建议取舍：

- 保留“首次句子到达”这次落盘，因为这是最关键的过程性数据。
- 保留“翻译完成 / 错误收口 / 声纹最终识别”这类最终态更新，但也应走节流队列，而不是即时整量写。
- 去掉 glossary pending、翻译 streaming delta 之类的中间态写盘。

收益：

- 同一条句子的本地快照写盘次数明显下降
- 翻译与声纹不再放大本地存储负担

### 4.3 实时转写区窗口化渲染

目标口径：

- 页面默认只保留最近 `100~300` 条可见气泡
- 更早内容只保留数据，不保留节点
- 用户上滑时，再补渲染更早的一段
- 用户回到底部时，再切回最新窗口

这一点必须明确：

- 只减 DOM，不减存储
- 不删除 `segmentStore`
- 不删除 `segmentOrder`
- 不减少已持久化的 transcript 数据
- 不影响恢复会议、总结、归档、replay

也就是说：

- 用户开超长会议后退出，再回来时，300 条以前的数据不能丢
- 300 条以前的数据仍应存在于内存恢复链路与持久化快照中
- 变化的只是“页面当前挂着哪些节点”，不是“会议真实保存了哪些数据”

## 5. 本轮不做什么

- 不重构 `meeting-store` 结构
- 不把 `localStorage` 迁移到 `IndexedDB`
- 不做 transcript chunk 分片存储
- 不改会议总结、归档、回放的数据结构
- 不改视觉样式与页面产品形态

## 6. 风险点

- 如果节流后没有在页面关闭 / 结束会议时强制 flush，最后一段 transcript 可能丢失。
- 如果窗口化渲染把“删除 DOM”误做成“删除 segment 数据”，会直接破坏恢复与归档。
- 如果恢复会议时只恢复最近窗口，没有恢复完整 transcript 数据，会导致 resume / archive / replay 不完整。
- 如果翻译与声纹的最终态也完全不落盘，恢复后页面会看不到对应最终结果。

## 7. 本轮验收口径

### 7.1 节流写盘

- 连续说话时，不再每条小更新都同步重写本地快照。
- 页面关闭前、结束会议前，会强制写入最后一版 transcript 快照。

### 7.2 减少重复写盘

- 翻译 streaming delta 不触发写盘。
- glossary / pending 中间态不触发整量写盘。
- 翻译 success / error、声纹最终识别仍可进入节流写盘队列。

### 7.3 窗口化渲染

- 实时转写默认只渲染最近一段气泡。
- 用户上滑可以继续看到更早内容。
- 长会议过程中 DOM 节点数不再无上限累计。

### 7.4 过程性存储不丢

- 开超长会议后退出，再进入同一会议时，旧 transcript 仍可恢复。
- 再次结束并归档后，replay 仍能看到完整历史，而不是只剩最近窗口。

## 8. 后续方向

如果后续要继续做长会议能力增强，推荐顺序如下：

1. `IndexedDB` 替代 `localStorage`
2. transcript chunk 化存储
3. 助手 / 总结上下文窗口化
4. 超长会议章节化或分段归档
5. 服务端持久化与多端恢复

## 9. 结论

P0 的核心原则是：

- 先减少同步整量写盘
- 先去掉中间态重复写盘
- 先把 DOM 从“无限增长”改成“窗口化渲染”

并且必须保证：

- 不破坏现有会议生命周期
- 不改变页面风格
- 不影响恢复会议、总结、归档与 replay
- 第三点只影响渲染，不影响过程性存储
